# 5장 안정 해시 설계

<aside> 💡

**서론**

수평적 규모 확장성을 달성하기 위해서는 **요청 또는 데이터를 서버에 균등하게 나누는 것**이 중요

이 목적을 위해 보편적으로 **안정 해시**를 사용함

그렇다면 이 해시 기술이 풀려고 하는 문제는 무엇일까?

</aside>

- 부가 정리
    - 대규모 시스템에서 **데이터를 여러 서버(샤드)에 나눠서 저장**해야 함 → 이게 **샤딩**
    - 이때, **“어떤 키(데이터)가 어떤 서버에 갈지”** 정하는 기술 중 하나 → 이게 **안정 해시**
    - 예시 (분산 시스템을 안 만들어 봐서 뭔가… key로 어떻게 데이터를 나누는지 와닿지 않아서 적어 놨음)
        - 채팅 서비스일 경우, 보통 사용자 ID/채팅방 ID를 키로 샤딩
        - 채팅방 ID = key → 해당 서버가 채팅방의 모든 메시지를 관리

## 5.1 해시 키 재배치(rehash)문제

**가정**: N개의 캐시 서버 존재

수평적 규모 확장성 달성을 위해 **요청 or 데이터를 서버에 균등하게 나누는 것**이 중요함

→ 이 서버들에 _**부하를 균등하게 나누는**_ 보편적 방법은? 아래의 **해시 함수**

<aside> 💡

**serverIndex = hash(key) % N (N은 서버의 개수)**

</aside>

그렇다면 예시를 통해 어떻게 동작하는지 알아보자

**가정**: 총 4개의 서버를 사용

**<주어진 각각의 키에 대해 해시 값, 서버 인덱스를 계산한 표>**

|키|해시|해시%4(서버 인덱스)|
|---|---|---|
|key0|18358617|1|
|key1|26143584|0|
|key2|18131146|2|
|key3|35863496|0|
|key4|34085809|1|
|key5|27581703|3|
|key6|38164978|2|
|key7|22530351|3|

위의 표에서 특정 키가 보관된 서버를 알아내기 위해, 나머지 연산을 f(key) % 4로 적용

![키값분배.png](attachment:e02ec4af-b9ae-4ea1-9afe-a4fcfc5a5043:%ED%82%A4%EA%B0%92%EB%B6%84%EB%B0%B0.png)

해당 그림을 통해 키 값이 어떻게 서버에 분산되는지 볼 수 있음

이 방법은 서버 풀(sever pool)의 크기가 고정된 경우, 데이터 분포가 균등할 땐 문제 X

**BUT, 서버 추가 or 기존 서버 삭제 시 문제 발생**

**예시** 1: 1번 서버가 장애를 일으켜 동작을 중단한 경우

⇒ 서퍼 풀의 크기가 3으로 줄어들게 됨

⇒ 키에 대한 해시 값은 동일하지만, 나머지 연산을 적용해 구한 **서버 인덱스 값은 달라짐**

|키|해시|해시%3(서버 인덱스)|

|key1|26143584|0|

|key6|38164978|1|

1번 서버에 보관된 키뿐만 아니라, **대부분의 키가 재분배되는 현상**이 발생함

⇒ 1번 서버가 죽었을 경우, 대규모 **캐시 미스**가 발생하게 될 것

- **대부분 캐시 클라이언트가 데이터가 없는 엉뚱한 서버에 접속**하게 됨을 의미

## 5.2 안정 해시

- **안정 해시**(consistent hash): 해시 테이블 크기가 조정될 때 평균적으로 **오직 k/n개의 키만 재배치하는** 해시 기술
    - k: 키의 개수, n: 슬롯(slot) 개수
    - 대부분의 전통적 해시 테이블: 슬롯의 수가 바뀌면 거의 대부분 키를 재배치 (이래서 문제다)

### 5.2.1 해시 공간과 해시 링

- 부가 정리
    
    키와 서버를 같은 해시 공간에 **‘모든 객체를 단순히 해시해서 놓기**’만 하면 자동 대응
    

<aside> 💡

그렇다면 이 안정 해시의 동작 원리는 어떻게 될까?

</aside>

**가정:** 해시 함수 f로 SHA-1를 사용, 출력 값의 범위는 x0, x1, x2, x3, .. xn

- SHA-1의 해시 공간(hash space) 범위: 0 ~ 2^160-1
- 따라서, x0 = 0, xn = 2^160-1, x1부터 xn-1까지는 범위 사이의 값

이 길다란 해시 공간을 양쪽을 구부려서 접으면 **해시 링(hash ring)**이 됨

![해시공간.png](attachment:888254a9-e192-44d5-bd46-2706ca0bd504:%ED%95%B4%EC%8B%9C%EA%B3%B5%EA%B0%84.png)

![해시링.png](attachment:f3d6f1a2-3965-4aec-aa7d-cccd262db053:%ED%95%B4%EC%8B%9C%EB%A7%81.png)

- 부가 정리: 해시 링은 어떤 구조?
    
    그림으로는 구부려서 링을 만든다는데 실제로는 그렇지 않을 것이라…
    
    - 실제 구현은 보통 **정렬된 배열, balanced tree, SortedMap(TreeMap)** → 논리적으로 이어졌다고 간주하는 **원형 구조**
        - 규칙을 만들어서 원형처럼 동작시키겠다는 의미
            
            <aside> 💡
            
            if (key보다 큰 서버 존재) → 그 서버가 담당 else → 첫 번째 서버가 담당 (원처럼 다시 처음으로 이어짐)
            
            </aside>
            
        - O(logn), O(1)에 가까운 속도라 빠름
            
    - 원형 리스트는 실제로 잘 안 쓴다고 함
        - 탐색 시간 O(n): 리스트를 싹 훑어야 함
        - 원형 리스트는 random access가 안 돼서 키/서버를 정렬된 순서로 유지하기 어려움
        - 대규모 분산 시스템과 맞지 않음

### 5.2.7 기본 구현법의 두 가지 문제

MIT에서 처음 제안된 안정 해시 알고리즘의 기본 절차는 아래와 같음

**기본 절차**

1. 서버와 키를 **균등 분포(uniform distribution) 해시 함수**를 사용해 해시 링에 배치함
2. 키의 위치에서 링을 시계 방향으로 탐색하다 만나는 최초의 서버가 키가 저장될 서버

문제점

1. 서버 추가/삭제 상황을 감안할 경우 파티션(partition)의 크기를 균등하게 유지하는 게 불가능함
    
    - 파티션: 인접한 서버 사이의 해시 공간
    - 부가 정리: 파티션 불균형이 왜 문제일까?
        - 파티션 크기 = 그 서버가 맡는 키의 범위
        - 파티션 크기 크면 = 많은 키 담당 → **부하 집중**
        - 파티션 크기 작으면 = 적은 키 담당 → **리소스 낭비**
        - 결론: 안정 해시는 ‘서버마다 키를 고르게 나누려고’ 쓰는 기술인데 파티션 크기가 불균형이면 → 서버가 받는 키 양도 불균형 → 부하 분산 실패
    
    ![문제1.png](attachment:f46e6bde-50a8-4a06-99ad-a3c5cecd6702:%EB%AC%B8%EC%A0%9C1.png)
    
2. 키의 균등 분포(uniform distribution)을 달성하기 어려움
    
    - 서버 1, 서버 3은 아무 데이터도 갖지 않지만, 서버 2는 대부분의 키가 몰빵됨
    - **서버 수가 적고 키는 랜덤**(해시값이 랜덤) → **불균형** 발생 (서버 추가/삭제와 무관한 통계적 현상)
    
    ![문제2.png](attachment:4286b279-d0a6-4f0f-8f07-d87096035e75:%EB%AC%B8%EC%A0%9C2.png)

### 5.2.8 가상 노드

**가상 노드(virtual node):** 실제 노드 or 서버를 가르키는 노드로, 하나의 서버는 링 위에 여러 개의 가상 노드를 가질 수 있음

**그림 5-12**

- 서버 0, 서버 1은 3개의 가상 노드를 가짐 (숫자 3은 임의로 정한 것으로, 실제 시스템은 훨씬 큰 값 사용)
- 따라서, 각 서버는 여러 개의 파티션을 관리해야 함
- s0은 서버 0, s1은 서버 1이 관리하는 파티션

![가상1.png](attachment:219c0899-83b6-4962-b78c-1ae14294b268:%EA%B0%80%EC%83%811.png)

**그림 5-13**

- 키의 위치로부터 시계방향으로 링을 탐색하다 만나는 최초의 가상 노드가 해당 키가 저장될 서버가 됨
- k0이 저장되는 서버는 가상 노드 s1.1을 제일 먼저 만나므로 서버 1이 됨

![가상2.png](attachment:9ea91f8d-a962-4a22-ad5e-0542f598754d:%EA%B0%80%EC%83%812.png)

**정리**

- **가상 노드의 개수를 늘리면** **표준 편차(stand-ard deviation)이 작아져 데이터가 고르게 분포**되기 때문에 **키의 분포는 더욱 균등**해질 것
    - 가상 노드로 **서버 수를 늘리는 것 같은 효과**를 줘서 **통계적 편차를 줄이고 키 분표를 균등화**하자는 의의?
- **표준 편차:** 데이터가 어떻게 퍼져 나갔는지 보이는 척도
- 100 ~ 200개의 가상 노드를 사용했을 경우, 표준 편차 값은 평균의 5%(200)에서 평균의 10%(100) 사이
- 가상 노드의 개수를 늘리면 표준 편차는 더 떨어지겠으나, **가상 노드 데이터 저장 공간은 더 많이 필요**해짐
- 따라서, **타협적 결정**(tradeoff)가 필요

## 5.3 마치며

**의미:** 안정 해시 필요 이유와 원리 이해

**장점**

- 서버 추가/삭제 → 재배치되는 키의 수 최소화

# 6장 키-값 저장소 설계

<aside> 💡

**서론**

**키-값 저장소(key-value store):** 키-값 데이터베이스라고도 불리는 비관계형(non-relational) 데이터베이스

EX) 아마존 다이나모, mencached, 레디스 등

- 이 저장소에 저장되는 값은 **고유 식별자**(identifier)를 **키**로 가져야 함
- “키-값” 쌍(pair): 키와 값 사이의 연결 관계
- 키-값 쌍에서 키는 **유일해야** 하며, 해당 키에 매달린 값은 **키를 통해서만 접근** 가능
- 키는 일반 텍스트일 수도, 해시 값일 수도
- **키는 짧을수록** 좋음(성능상의 이유)
- 값은 문자열일 수도, 리스트일 수도, 객체일 수도 → 키-값 저장소는 **값으로 무엇이 와도 상관 X** </aside>

<aside> 💡

**가정:** 다음 연산을 지원하는 키-값 저장소를 설계

- put(key, value): 키-값 쌍을 저장소에 저장
- get(key): 인자로 주어진 키에 매달린 값을 꺼냄 </aside>

## 6.1 문제 이해 및 설계 범위 확정

완벽한 설계는 없으므로 타협적 결정을 내린 설계라면 쓸만한 답일 것

다음 특성을 갖는 키-값 저장소를 설계할 것

- 키-값 쌍의 크기는 10KB 이하
- 큰 데이터 저장 가능
- 높은 가용성 제공 → 시스템은 설사 장애가 있어도 빨리 응답해야 함
- 높은 규모 확장성 제공 → 트래픽 양에 따라 자동적으로 서버 증설/삭제
- 데이터 일관성 수준은 조정이 가능해야 함
- 응답 지연시간(latency)이 짧아야 함

## 6.2 단일 서버 키-값 저장소

서버 한 대 사용하는 키-값 저장소는 설계하는 것은 쉬움

가장 직관적인 방법: 키-값 쌍 전부를 메모리에 해시 테이블로 저장하는 것

**문제점**: 빠른 속도가 보장되지만 모든 데이터를 메모리에 두는 것은 불가능할 수 있음

**개선책**

- 데이터 압축
- 자주 쓰이는 데이터만 메모리에 두고 나머지는 디스크에 저장

⇒ 하지만, 한 대 서버로 부족할 때가 찾아오므로, **많은 데이터를 저장하기 위해 분산 키-값 저장소**(distributed key-value store)를 만들 필요가 있음

## 6.3 분산 키-값 저장소

분산 키-값 저장소는 **분산 해시 테이블**이라고도 하는데, **키-값 쌍을 여러 서버에 분산시키기 때문**임

분산 시스템을 설계할 땐 먼저 **CAP 정리**를 이해해야 함

### 6.3.1 CAP 정리

**CAP 정리:** **데이터 일관성**(consistency), **가용성**(availability)**, 파티션 감내**(partition tolerance) 같은 세 가지 요구사항을 **동시에 만족하는 분산 시스템**을 설계하는 것은 **불가능**하다는 정리

- **데이터 일관성:** 분산 시스템에 접속하는 모든 클라이언트들은 **어떤 노드에 접속했느냐에 관계없이 언제나 같은 데이터를 보게 되어야** 함
- **가용성:** 분산 시스템에 접속하는 클라이언트는 **일부 노드에 장애가 발생하더라도 항상 응답을 받을 수 있어야 함**
- **파티션 감내:** 파티션은 두 노드 사이에 통신 장애가 발생하였음을 의미 → 파티션 감내는 네트워크에 **파티션이 생기더라도 시스템은 계속 동작하여야 함**을 의미
- CAP 정리는 두 개를 충족하면 반드시 하나는 희생되어야 함을 뜻함

키-값 저장소는 **충족하는 두 가지의 종류에 따라 분류**할 수 있음

- **CP 시스템**: 일관성, 파티션 감내 O, 가용성 X
- **AP 시스템:** 가용성, 파티션 감내 O, 일관성 X
- **CA 시스템**: 일관성, 가용성 O, 일관성 X

⇒ 사례로 위의 정의를 이해해 보자

### 6.3.2 시스템 컴포넌트

키-값 저장소에 구현에 사용될 **핵심 컴포넌트와 기술**을 알아보자

**목록**

- 데이터 파티션
- 데이터 다중화
- 일관성
- 일관성 불일치 해소
- 장애 처리
- 시스템 아키텍처 다이어그램
- 쓰기 경로
- 읽기 경로

해당 내용은 널리 사용 중인 **다이나모(Dynamo), 카산드라(Cassandra), 빅테이블(BigTable) 사례를 참고**한 것

### 6.3.2.1 **데이터 파티션**

대규모 애플리케이션 → 전체 데이터를 한 서버에 저장하는 것은 불가능

**가장 단순한 해결책**: 데이터를 작은 파티션으로 분할 → 여러 대의 서버에 나눠서 저장

**데이터를 파티션 단위로 나눌 때 따져야 할 두 가지 문제**

1. 데이터를 여러 서버에 **고르게 분산**할 수 있는가?
2. 노드 추가/삭제 시 데이터의 **이동을 최소화**할 수 있는가?

⇒ 5장의 안정 해시(consistent hash)가 이 문제를 풀 때 적합한 기술

**안정 해시로 데이터를 파티션할 때 장점**

1. **규모 확장 자동화**(automatic scailing): 시스템 부하에 따라 서버가 자동으로 추가/삭제 가능
2. **다양성**(heterogeneity): 각 서버의 용량에 맞게 가상 노드 수 조정 가능 → 고성능 서버에 더많은 가상 노드를 부여

### 6.3.2.2 **데이터 다중화(replication)**

높은 **가용성, 안정성 확보**를 위해 데이터를 N개의 서버에 **비동기적으로 다중화**(replication)할 필요 있음 (동기 다중화라면 모든 복제본에 쓰기가 완료될 때까지 대기할 것이기 때문에 분산 시스템에서 느릴 것)

- N: 튜닝 가능한 값
    
- N개 서버를 선정하는 법
    
    1. 어떤 키를 해시 링 위에 배치
    2. 그 지점으로부터 시계 방향으로 링을 순회하면서 만나는 **첫 n개의 서버에 데이터 사본 보관**
    
    - **그림 6-5:** key0(데이터)는 시계 방향으로 s1, s2, s3에 저장
    - **문제:** 가상 노드 사용 시 선택한 N개의 노드가 대응될 실제 물리 서버의 개수가 N보다 작아질 수 있음
    - **해결:** 노드를 선택할 때 같은 물리 서버를 중복 선택하지 않게 함
        - ex) 서울 데이터 센터에 데이터를 넣었다면 타 리전에 저장을 합시다
    
    ![65.png](attachment:0aad9cde-7da2-4a8a-add9-d0216f93b8ed:65.png)
    

같은 데이터 센터에 속한 노드: 정전, 네트워크 이슈, 자연재해 등의 문제를 **동시에 겪을 가능성 O**

⇒ 안정성을 위해 데이터 **사본은 다른 센터** 서버에, 센터들은 **고속 네트워크**로 연결

### 6.3.2.3 **데이터 일관성(consistency)**

여러 노드에 다중화된 데이터는 **적절한 동기화** 필요

**정족수 합의**(Quorun Consensus) 프로토콜을 통해 읽기/쓰기 연산 모두 **일관성 보장** 가능

**관계된 정의**

- N = 사본 개수
    
- W = 쓰기 연산에 대한 정족수(quorum) → 쓰기 연산이 성공한 것으로 간주되려면 적어도 W개의 서버로부터 쓰기 연산이 성공했다는 응답을 받아야 함
    
- R = 읽기 연산에 대한 정족수 → 읽기 연산이 성공한 것으로 간주되려면 적어도 R개의 서버로부터 응답을 받아야 함
    
- **N=3일 때 그림 6-6**
    
    ![66.png](attachment:591988bd-451b-4179-b4d1-9e533eb83d24:66.png)
    
    - `W = 1`: 쓰기 연산이 성공했다고 판단하기 위해 중재자(coordinator)는 최소 1대 서버로부터 쓰기 응답을 받아야 함
        - s1에서 응답을 받았다면, s0, s2 응답 필요 X
        - 중재자는 클라이언트-노8드 사이의 **프록시**(proxy) 역할
    
    <aside> 💡
    
    W, R, N의 값을 정하는 것 = 응답 지연과 데이터 일관성 사이의 타협점을 찾는 전형적 과정
    
    </aside>
    
- `W = 1`, `R = 1`: 중재자는 1대 서버만 응답을 받으면 되니 응답속도는 빠를 것
    
- `W > 1`, `R > 1`: 시스템이 보여 주는 데이터 일관성 수준은 올라감, 중재자 응답 속도는 확인해야 하는 서버 중 가장 느린 서버로부터 응답을 기다려야 함으로 느려질 것
    
- `W + R > N`: **강한 일관성**(strong consistency) 보장됨
    
    - 일관성을 보증할 최신 데이터를 가진 노드가 **최소 하나는 겹칠 것**(두 개 합친 게 사본 개수보다 많으니까?)

**정리: N, W, R 값은 어떻게 정할까?**

- `R = 1`, `W = N`: 빠른 읽기 연산 최적화
- `W = 1`, `R = N`: 빠른 쓰기 연산 최적화
- `W + R > N`: 강한 일관성 보장 (보통 N=3, W=R=2) ← 왜 보통 이 값일까?
- `W + R <= N`: 강한 일관성 보장 X

### 6.3.2.4 **일관성 불일치 해소(inconsistency resolution)**

**일관성 모델**(consistency model): 키-값 저장소를 설계할 때 고려해야 할 중요 요소 중 하나로, 데이터의 일관성 수준을 결정하며 종류가 다양함

- **강한 일관성**(strong consistency): 모든 읽기 연산은 가장 최근에 갱신된 결과 반환
    
    ⇒ 모든 클라이언트는 절대로 낡은(out-of-data) 데이터 못 봄
    
    - 모든 사본에 현재 쓰기 연산 결과가 반영되기 전까지 해당 데이터의 읽기/쓰기 금지
        
        ⇒ 고가용성 시스템에 부적절, 새로운 요청 처리가 중단되기 때문
        
- **약한 일관성**(weak consistency): 읽기 연산은 가장 최근 갱신된 결과 반환 못할 수 있음 ⇒ 최신 데이터를 못 본다는 정도의 매우 넓은 개념(개념 자체가 느슨)
    
- **최종 일관성**(eventual consistency): 약한 일관성의 한 형태로 갱신 결과가 결국 모든 사본에 반영(동기화)되는 형태
    
    - 다이나모, 카산드라 케이스 (이번 장도 해당 케이스로 설계할 것)
        
    - 이 모델을 따를 경우 쓰기 연산이 병렬적으로 발생하면 시스템에 저장된 값의 일관성이 깨질 수 있음
        
        ⇒ 이건 클라이언트가 해결해야 함
        
        ⇒ 약한 일관성과 다르게 구체적이라 실제 구현에서 실제로 충돌이 발생하는 걸 알 수 있으니까 구체적인 사례에 대한 해결책을 알아보는 것
        
        <aside> 💡
        
        클라이언트 측에서 데이터 버전 정보를 활용해 깨진 데이터를 읽지 않도록 하는 기법에 대해 살펴보자
        
        </aside>

**해결**: **버저닝**(versioning)과 **벡터 시계**(vector clock)

**벡터 시계**

[서버, 버전]의 순서쌍을 데이터에 매단 것

어떤 버전이 선행 버전인지 후행 버전인지 아니면 다른 버전과 충돌이 있는지 관찰할 때 쓰임

- D([S1, v1], [S2, v2], [S3, v3], …, [Sn, vn]처럼 벡터 시계를 표현한다고 가정
    
- D = 데이터, vi = 버전 카운터, Si = 서버 번호
    
- IF 데이터 D를 서버 Si에 기록하면 시스템은 아래 작업 중 하나를 수행해야 함
    
    - [Si, vi]가 있으면 vi 증가
    - 그렇지 않으면 새 항목 [S1, 1] 생성
    
    ![69.png](attachment:51fb6cff-0f53-4c1a-8c10-cbf3dcb02467:69.png)
    
    **구체적 사례**
    
    1. 클라이언트가 데이터 D1을 시스템에 기록
        - 쓰기 연산은 Sx가 처리→ 벡터 시계는 D1([Sx, 1])로 변화
    2. 다른 클라이언트가 데이터 D1을 읽고 D2로 업데이트한 다음 기록
        - D2는 D1에 대한 변경이므로 D1을 덮어씀
        - 쓰기 연산은 Sx가 처리 → 벡터 시계는 D1([Sx, 2])로 변화
    3. 다른 클라이언트가 D2를 읽고 D3으로 갱신한 다음 기록
        - 쓰기 연산은 Sy가 처리 → 벡터 시계는 D3([Sx, 2], [Sy, 1])로 변화
    4. 또 다른 클라이언트가 D2를 읽고 D4로 갱신한 다음 기록
        - 쓰기 연산은 Sz가 처리 → 벡터 시계는 D4([Sx, 2], [Sz, 1])로 변화
    5. 어떤 클라이언트가 D3과 D4를 읽으면 데이터 충돌이 있다는 것을 인지
        - D2를 Sy와 Sz가 각기 다른 값으로 바꾸었기 때문
            
            ⇒ 이 충돌을 클라이언트가 해소하고 서버에 기록
            
        - 쓰기 연산은 Sx가 처리 → 벡터 시계는 D5([Sx, 3], [Sy, 1], [Sz, 1])로 변화
            
            ⇒ 충돌 감지는 어떻게 하는지 뒤에 설명
            

**결론**

벡터 시계를 통해 어떤 버전 X가 버전 Y의 **이전 버전인지(따라서 충돌이 없는지) 쉽게 판단** 가능

- 버전 Y에 포함된 모든 구성요소의 값 ≥ X에 포함된 모든 구성요소의 값
- EX D([s0, 1], [s1, 2])와 D([s0, 2], [s1, 1])은 서로 충돌

**단점**

1. 충돌 감지 및 해소 로직이 클라이언트에 들어가야 함 → 클라이언트 구현 복잡해짐
    
2. [서버:버전]의 순서쌍 개수가 굉장히 빨리 늘어남
    
    ⇒ 해결 위해 그 길이에 어떤 임계치(threshold)를 설정, 임계치 이상으로 길이가 길어지면 오래도니 순서쌍을 제거
    
    ⇒ 근데 이러면 버전 간 선후 관계가 정확히 결정될 수 없어 충돌 해소 과정 효율성이 낮아짐
    
    - 근데 다이나모 데이터베이스 관계 문헌 기준 아마존은 그런 문제 벌어지는 것을 못 봤다고 함 (그러니까 괜찮다고 생각해도 됨)

### **장애 처리**

대다수 대규모 시스템에서 장애는 아주 흔하게 벌어지는 사건

⇒ 따라서 **장애를 어떻게 처리할 것이냐**는 중요한 문제

<aside> 💡

두 가지를 알아보자

1. **우선 장애 감지**(failure detection) 기법
2. **장애 해소**(failure resolution) 전략 </aside>

**장애 감지**

보통 분산 시스템에서는 **두 대 이상의 서버가 똑같이 서버 A의 장애를 보고하는 상황**에 서버가 실제로 장애가 발생했다고 간주

- 방법 1: 모든 노드 사이에 **멀티캐스팅** 채널 구축
    - 서버가 많을 때 비효율적
- 방법 2: **가십 프로토콜 같은 분산형 장애 감지 솔루션** 채택
    - **가십 프로토콜 동작 원리**
        1. 각 노드는 멤버십 목록 유지 (멤버십 목록: 각 멤버 ID와 그 박동 카운터 쌍의 목록)
        2. 각 노드는 주기적으로 자신의 박동 카운트 증가시킴
        3. 각 노드는 무작위로 선정된 노드들에게 주기적으로 자기 박동 카운터 목록 보냄
        4. 박동 카운터 목록을 받은 노드는 멤버십 목록을 최신 값으로 갱신
        5. 어떤 멤버의 박동 카운터 값이 지정된 시간 동안 갱신 X ⇒ 해당 멤버가 장애(Offline) 상태로 간주

**일시적 장애 처리 ⇒ 단서 후 임시 위탁 기법 사용**

가십 프로토콜로 장애를 감지한 시스템은 **가용성 보장을 위해** 필요한 조치를 해야 함

- **엄격한 정족수**(strict quorum) 접근법: 읽기/쓰기 연산 모두 금지
- **느슨한 정족수**(sloppy quorum) 접근법: 위 조건을 완화해 가용성을 높임
    - 정족수 요구사항은 강제 (아래의 내용이 느슨한 정족수 한정인지 필요한 조치 전체에 대한 내용인지?)
        
    - 쓰기 연산을 수행할 W개의 건강한 서버와 읽기 연산을 수행할 R개의 건강한 서버를 해시 링에서 선택
        
    - 장애 상태 서버 무시 → 이 서버로 가는 요청은 다른 서버가 임시로 처리
        
    - 그동안 발생한 변경사항은 해당 서버 복구 시점에 일괄 반영하여 데이터 일관성 보존
        
    - 임시로 쓰기 연산을 처리한 서버에 그에 대한 단서를 남김 → 해당 장애 처리 방안을 **단서 후 임시 위탁**(hinted handoff) 기법이라 부름
        
        [https://cassandra.apache.org/doc/latest/cassandra/managing/operating/hints.html](https://cassandra.apache.org/doc/latest/cassandra/managing/operating/hints.html)
        

**영구 장애 처리**

<aside> 💡

그렇다면 영구적인 노드 장애 상태는 어떻게 처리해야 할까?

**반-엔트로피(anti-entropy) 프로토콜**을 구현하여 사본들을 **동기화**해야 함

</aside>

- 반-엔트로피 프로토콜은 사본들을 비교해 최신 버전으로 갱신하는 과정을 포함함
    
- 사본 간의 일관성이 망가진 상태를 탐지하고 전송 데이터 양을 줄이기 위해 ⇒ 머클(Merkle) 트리 사용 (사본의 서로 다른 부분을 빠르게 찾는 기술로 채택하겠다)
    
- 부가 정리: 머클 트리는 뭘까?
    
    **각 계층의 노드가 “하위 노드들의 해시 값”을 저장하는 트리 구조**
    
    - 리프 노드: “데이터 자체 해시 값”
    - 부모 노드: “자식 두 개의 해시 값”을 다시 해싱한 값
    - 루트 노드: “전체 데이터 묶음에 대한 해시 값”
    
    **해시 → 데이터가 조금이라도 변하면 값이 크게 변함**
    
    ⇒ 따라서 “같은 루트 해시면 전체 데이터가 동일하다는 증거”
    
    [https://terms.tta.or.kr/dictionary/dictionaryView.do?word_seq=171397-2](https://terms.tta.or.kr/dictionary/dictionaryView.do?word_seq=171397-2)
    

**머클(Merkle) 트리**: 해시 트리라고도 불리는 머클 트리는 각 노드에 그 자식 노드들에 보관된 값의 해시(자식 노드가 종단 leaf 노드인 경우), 자식 노드들의 레이블로부터 계산된 해시 값을 레이블로 붙여두는 트리

**분산 키-값 저장소가 가져야 하는 기능 및 기능 구현에 이용되는 기술**

|**목표/문제**|**기술**|
|---|---|
|대규모 데이터 저장|안정 해시로 서버 부하 분산 해결|
|읽기 연산 높은 가용성|데이터를 여러 데이터 센터에 다중화|
|쓰기 연산 높은 가용성|버저닝 및 벡터 시계를 이용한 충돌 해소|
|데이터 파티션|안정 해시|
|점진적 규모 확장성|안정 해시|
|다양성|안정 해시|
|조절 가능 데이터 일관성|정족수 합의|
|일시적 장애 처리|느슨한 정족수 프로토콜과 단서 후 임시 위탁|
|영구적 장애 처리|머클 트리|
|데이터 센터 장애 대응|여러 데이터 센터에 걸치 데이터 다중화|
