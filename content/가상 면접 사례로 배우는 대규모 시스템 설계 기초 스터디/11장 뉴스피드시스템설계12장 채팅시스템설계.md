# **11. 뉴스 피드 시스템 설계 (News Feed System Design)**

뉴스 피드 시스템 설계는 페이스북, 인스타그램 피드 또는 트위터 타임라인 설계와 유사한 유명한 면접 문제

### **1단계: 문제 이해 및 설계 범위 확정**

- **뉴스 피드의 정의:** 사용자의 홈 페이지 중앙에 지속적으로 업데이트되는 스토리(상태 업데이트, 사진, 비디오, 좋아요 등)를 포함
- **주요 기능:** 사용자는 새로운 스토리를 올릴 수 있어야 하고, 친구들이 올린 스토리를 볼 수 있어야 합니다. 스토리에는 이미지나 비디오 등의 미디어 파일이 포함
- **규모 및 순서:** 시스템은 매일 천만 명(10 million DAU)의 트래픽을 처리해야 하며, 한 사용자는 최대 5,000명의 친구를 가질 수 있습니다. 스토리는 단순히 시간 흐름 역순(reverse chronological order)으로 표시된다고 가정

### **3단계: 상세 설계 – 피드 발행 흐름 (팬아웃)**

![image.png](attachment:24eab054-aff4-4a05-8489-bde631fcc824:image.png)

**피드 발행 시 친구들에게 새 포스팅을 전달하는 과정**을 **포스팅 전송(팬아웃, fanout)**이라고 함

1. **쓰기 시점 팬아웃 (Fanout-on-Write, 푸시 모델):** 새로운 포스팅이 기록되는 시점에 바로 친구들의 뉴스 피드를 갱신
    - 장점: 뉴스 피드가 실시간으로 갱신되며, 읽는 시간이 짧다(pre-computed).
    - 단점: 친구가 많은 사용자(핫키)의 경우 갱신에 시간이 오래 걸리고, 비활성화된 사용자 피드까지 갱신하여 컴퓨팅 자원이 낭비될 수 있다
2. **읽기 시점 팬아웃 (Fanout-on-Read, 풀 모델):** 피드를 읽어야 하는 시점(요청 기반, on-demand)에 뉴스 피드를 갱신
    - 장점: 비활성화된 사용자에게는 컴퓨팅 자원을 소모하지 않습니다. 핫키 문제가 생기지 않음
    - 단점: 뉴스 피드를 읽는 데 시간이 오래 걸릴 수 있다

**선택된 전략 (하이브리드)**

- 뉴스 피드를 빠르게 가져오는 것이 중요하므로, **대부분의 사용자에게는 푸시 모델**을 사용
- 다만, 친구나 팔로어가 **아주 많은 사용자(핫키 사용자)의 경우**에는 시스템 과부하를 막기 위해 **풀 모델**을 사용하도록 결합
- 핫키 문제를 줄이기 위해 안정 해시(consistent hashing)도 사용

**팬아웃 서비스 동작:**

![image.png](attachment:17f9f98e-17ee-4f8b-8856-3d74afff17df:image.png)

1. 그래프 데이터베이스에서 친구 ID 목록을 가져옴
    - [그래프 데이터 베이스](https://www.oracle.com/kr/autonomous-database/what-is-graph-database/)
        - 친구 관계나 친구 추천을 관리하기 용이
        - Facebook’s Distributed Data Store for the Social Graph 논문https://www.usenix.org/system/files/conference/atc13/atc13-bronson.pdf
            
            - **그래프 DB의 특징**: 노드(Node)와 엣지(Edge)로 관계를 표현
            - **언제 쓰나**: 친구 관계, 추천 시스템, 소셜 네트워크, 권한 관리 등
            - **왜 RDBMS 대신?**: 다단계 관계 조회(친구의 친구의 친구)가 JOIN보다 빠름
            - 예시 : Neo4j
            - Facebook(TAO): RDBMS + 캐싱 레이어로 그래프 표현하여 사용
            - 적은 규모: RDBMS의 재귀 쿼리(CTE) 활용하는 것을 추천
            
            1. 모니터링 지표
                - 캐시 히트율 (TAO는 96.4%)
                - 레플리케이션 지연 (TAO는 85%가 1초 미만)
                - P99 레이턴시
            2. 장애 대응
                - 리더 장애 시 팔로워가 DB 직접 접근
                - 마스터 DB 장애 시 슬레이브 승격
                - 부분 장애 허용 설계
            3. 확장성
                - Sharding 전략 (TAO의 샤드 설계)
                - 읽기/쓰기 분리
                - 지역별 복제
2. 사용자 설정(예: 뮤트 설정)에 따라 친구 중 일부를 거름
3. 친구 목록과 새 스토리의 포스팅 ID를 메시지 큐에 넣음
4. 팬아웃 작업 서버가 메시지 큐에서 데이터를 꺼내어 **뉴스 피드 캐시**에 `<포스팅 ID, 사용자 ID>` 순서쌍 형태로 저장
    - 어떤 사용자가 뉴스 피드에 올라온 수 천 개의 스토리를 전부 훑어보는 일이 벌어질 확률은 극히 낮고, 최신 스토리만 확인하고자 하므로 캐시 미스가 일어날 확률이 낮다

### **3단계: 상세 설계 – 피드 읽기 흐름 및 캐시 구조**

- **피드 읽기 흐름**
    
    ![image.png](attachment:5abeebb7-0c56-4224-9e16-6003c7f65a18:image.png)
    
    - 클라이언트가 요청하면 웹 서버를 거쳐 뉴스 피드 서비스가 호출
    - 뉴스 피드 서비스는 **뉴스 피드 캐시**에서 포스팅 ID 목록을 가져온 뒤, 사용자 캐시와 포스팅 캐시에서 이름, 사진, 콘텐츠 등의 상세 정보를 가져와 완전한 뉴스 피드를 JSON 형태로 클라이언트에게 보냄
    - 이미지나 비디오 같은 미디어 콘텐츠는 CDN에 저장하여 빠르게 읽어갈 수 있도록 함
- **캐시 구조 (5계층)**
    
    - 뉴스 피드, 콘텐츠(인기 콘텐츠/일반 콘텐츠), 소셜 그래프, 행동(좋아요, 답글), 횟수(좋아요 횟수 등)의 5계층으로 캐시를 구성

## **4단계: 마무리**

### 데이터베이스 규모 확장

- 수직적 규모 확장 VS 수평적 규모 확장
- SQL vs NoSQL
- 복제본(replica)에 대한 읽기 연산
- 일관성 모델(consistency model)
- 데이터 베이스 샤딩(sharding)

### 기타

- **웹 계층(webtier)을 무상태로 운영하기**
    
    aws [https://ga0lee.tistory.com/entry/AWS-3-Tier-Architecture-Web-Application-Architecture](https://ga0lee.tistory.com/entry/AWS-3-Tier-Architecture-Web-Application-Architecture)
    
    상태(State)를 웹 서버에 저장하지 않고, 외부 저장소(Redis, DB)에 저장 → 어떤 웹 서버로 요청이 가도 동일하게 처리 가능
    
    - 웹 서버를 언제든 추가/제거할 수 있어서 확장이 쉬움
        - 사용자 세션을 특정 서버에 고정(sticky session)하면 그 서버가 죽으면 세션 손실
        - 무상태면 로드밸런서가 자유롭게 트래픽 분산 가능
    - 사용자 인증 토큰만 받아서 Redis에서 세션 조회
    - 웹 서버는 요청만 처리하고 상태는 Redis/DB에 저장
- **가능한 한 많은 데이터를 캐시할 방법**
    
    자주 읽히는 데이터를 메모리(Redis, Memcached)에 저장 → DB 부하 감소 + 응답 속도 향상
    
    ⇒ 뉴스피드의 5계층 캐시 전략, 문서에 나온 것처럼 계층별로 캐시를 나눠서 관리
    
    1. Cache-Aside (Lazy Loading)
        - 읽을 때만 캐시에 로드
        - 가장 흔한 패턴
    2. Write-Through
        - 쓸 때 캐시도 함께 업데이트
        - 뉴스피드의 팬아웃에서 사용
    3. Write-Behind
        - 캐시에 쓰고 나중에 DB에 비동기로 저장
        - 좋아요 카운트에 적합
- 여러 데이터 센터를 지원할 방법
    
    지리적으로 분산된 데이터 센터를 운영 → 사용자와 가까운 곳에서 서비스 제공 (지연시간 감소)
    
- 메시지 큐를 사용하여 컴포넌트 사이의 결합도 낮추기
    
    직접 호출하지 않고 메시지 큐를 통해 비동기 처리 → 컴포넌트 간 의존성 감소 + 확장성 향상
    
    팬아웃 서비스가 대표적인 예시
    
    ```
    [동기 방식 - 나쁜 예]
    사용자 포스팅 → 웹서버 → (5000명 친구에게 일일이 전송) → 완료까지 10초 걸림
    
    [비동기 방식 - 좋은 예]
    사용자 포스팅 → 웹서버 → 메시지 큐에 넣음 → 즉시 응답 (0.1초)
                                ↓
                             팬아웃 워커가 백그라운드에서 처리
    
    ```
    
    ```
    [실제예시]
    
    ┌─────────────┐
    │  웹 서버    │ → 포스팅 저장
    └──────┬──────┘
           │
           ↓ (메시지 발행)
    ┌─────────────────────┐
    │  메시지 큐 (Kafka)  │
    │  Topic: feed_fanout │
    └──────┬──────────────┘
           │
           ↓ (메시지 소비)
    ┌──────────────────┐
    │ 팬아웃 워커 (x10)│ → 친구들 피드 캐시에 저장
    └──────────────────┘
    ```
    
- 핵심 메트릭(key metric)에 대한 모니터링.
    
    - 예를 들어 트래픽이 몰리는 시간대 의 QPS(Queries per Second)
    - 사용자가 뉴스 피드를 새로고침 (refresh) 할 때의 지연시간
    
    시스템 상태를 실시간으로 측정하고 알림 → 장애 예방 및 빠른 대응
    
    → QPS, 지연시간, 캐시 히트율, 에러율 → Grafana(그라파나) + Prometheus(프로메테우스)

# **12. 채팅 시스템 설계 (Chat System Design)**

이 장에서는 페이스북 메신저와 유사한 채팅 앱을 설계

- 1:1 채팅에 집중하는 앱
    
- 그룹 채팅에 중점을 둔 업무용 앱
    
- 대규모 그룹의 소통과 응답지연이 낮은 음성 채팅에 집중하는 앱
    
    등이 있으므로 문제 이해 및 설계 범위 확정 시 고려해봐야 한다
    

### **1단계: 문제 이해 및 설계 범위 확정**

- **주요 기능:** 응답 지연이 낮은 1:1 채팅, 최대 100명까지 참여 가능한 그룹 채팅, 사용자 접속 상태 표시, 다양한 단말 지원 및 푸시 알림을 지원
- **규모:** 일별 능동 사용자 수(DAU) 5천만 명(50 million)을 처리 가능
- **데이터 보존:** 채팅 이력은 영원히 보관

### **2단계: 개략적 설계안 및 통신 프로토콜**

- **메시지 송신 프로토콜:** 클라이언트에서 서버로 메시지를 보낼 때는 HTTP 프로토콜이 효율적일 수 있음
- **메시지 수신 프로토콜 (핵심):** 서버가 클라이언트에게 임의 시점에 메시지를 비동기적으로(async) 보내야 하므로, **웹소켓(WebSocket)**이 가장 널리 사용됨
    - 웹소켓은 클라이언트가 시작하며, 특정 핸드셰이크 절차를 거쳐 변화하지 않고 오래 지속되며 **양방향(bidirectional) 통신이 가능한 연결**로 업그레이드 됨
    - 비교: 폴링(Polling)이나 롱 폴링(Long Polling)은 메시지가 없어도 주기적으로 접속을 시도하거나 타임아웃을 기다려야 하므로 비효율적

**개략적 설계안 구성**

![image.png](attachment:dd914a02-1272-4042-858c-90c3b887cc9a:image.png)

1. **무상태 서비스 (Stateless):** 로그인, 회원가입, 프로파일 관리 등 전통적인 요청/응답 기능을 처리(주로 HTTP 사용).
2. **상태 유지 서비스 (Stateful):** 클라이언트와 웹소켓 연결을 유지해야 하는 **채팅 서버**와 **접속상태 서버(Presence Server**)가 해당.
3. **제3자 서비스 연동:** 푸시 알림 서비스와의 통합이 중요

### **데이터 모델 및 저장소**

- **데이터 유형:** 일반적인 데이터(프로파일, 친구 목록)는 관계형 DB에 보관하며, **채팅 이력(Chat History)**은 키-값 저장소(Key-Value Store)에 보관하는 것을 추천
    - **키-값 저장소 채택 이유:** 수평적 규모 확장(horizontal scaling)이 쉽고, 데이터 접근 지연시간(latency)이 낮으며, 오래된 데이터(롱 테일) 처리에도 용이하기 때문. (예: 페이스북 메신저는 HBase 사용).
- **메시지 ID:** 메시지 ID는 **고유해야 하며 시간 순서와 일치하도록 정렬 가능**. 구현이 쉬운 **지역적 순서 번호 생성기(Local Sequence Number Generator)**를 사용하여 해당 채널이나 1:1 세션 내에서만 유일성을 보증하는 방식으로 진행

### **3단계: 상세 설계 – 핵심 기능**

1. **서비스 탐색 (Service Discovery):**
    
    ![image.png](attachment:869c0abb-0563-4fdc-adf4-9dc014c605df:image.png)
    
    - 클라이언트에게 **가장 적합한 채팅 서버**를 추천하는 기능 (지리적 위치, 서버 용량 기준).
    - 클라이언트 로그인 시 API 서버가 인증을 처리하고, 서비스 탐색 기능(예: 주키퍼)이 최적의 채팅 서버를 찾아 반환하며, 클라이언트는 그 서버와 웹소켓 연결을 맺음.
2. **메시지 흐름**
    
    1. **1:1 채팅 메시지 처리 흐름**
        
        1. **메시지 전송**: 사용자 A → 채팅 서버 1로 메시지 전송
        2. **ID 생성**: 채팅 서버 1이 ID 생성기를 통해 메시지 ID 결정
        3. **큐 전송**: 메시지를 메시지 동기화 큐로 전송
        4. **저장**: 메시지가 키-값 저장소에 보관
        5. **수신자 처리**:
            - **(a) 접속 중인 경우**: 사용자 B가 접속한 채팅 서버(채팅 서버 2)로 메시지 전송
            - **(b) 미접속인 경우**: 푸시 알림 서버로 푸시 알림 메시지 전송
        6. **최종 전달**: 채팅 서버 2가 웹소켓 연결을 통해 사용자 B에게 메시지 전송
    2. **여러 단말 사이의 메시지 동기화**
        
        - 사용자 A가 전화기와 랩톱 두 대의 단말 사용
        - 각 단말은 채팅 서버 1과 별도의 웹소켓 연결 유지
        
        **동기화 메커니즘**
        
        **`cur_max_message_id` 변수**
        
        - **목적**: 각 단말에서 관측된 가장 최신 메시지 ID 추적
        - **관리**: 단말별로 독립적으로 유지
        
        **새 메시지 판별 조건**
        
        1. 수신자 ID = 현재 로그인한 사용자 ID
        2. 메시지 ID > `cur_max_message_id` (키-값 저장소 기준)
    3. **소규모 그룹 채팅에서의 메시지 흐름**
        
        **발신 흐름**
        
        - 사용자 A가 그룹 채팅방(3명: A, B, C)에 메시지 전송
        - 메시지가 사용자 B와 C의 **메시지 동기화 큐**에 각각 복사됨
        - 각 사용자의 큐 = 개인별 메시지 수신함
        
        **수신 흐름**
        
        - 각 수신자는 여러 사용자로부터 메시지 수신 가능
        - 메시지 동기화 큐는 다중 발신자로부터의 메시지 수용
        
        **소규모 그룹에 적합한 이유**
        
        |장점|설명|
        |---|---|
        |**단순한 동기화**|각 사용자가 자신의 큐만 확인하면 되므로 메시지 동기화 플로가 간단함|
        |**낮은 비용**|그룹이 작으면 메시지 복사 작업의 비용이 문제되지 않음|
        
3. **접속 상태 관리 (Presence):**
    
    - 접속 상태 서버(presence server)가 사용자의 상태를 관리
        
        - 로그아웃 시 키-값 저장소에 보관된 사용자 상태가 online→ offline으로 바뀌게 된다
    - **접속 장애 처리**
        
        - 짧은 인터넷 연결 끊김에도 상태가 자주 바뀌는 것을 방지하기 위해 **박동(Heartbeat) 검사**를 사용
        - 클라이언트가 주기적으로 박동 이벤트를 접속 상태 서버로 보내고, 일정 시간(X초) 동안 박동 이벤트가 없어야 오프라인으로 상태를 변경
    - **상태 정보 전송**
        
        ![image.png](attachment:a7aa5842-8523-43ae-a9c1-ebea064bf28f:image.png)
        
        - 친구 관계에 있는 사용자들에게 상태 변화를 알리기 위해 **발행-구독 모델 (publish-subscribe model)**을 사용
        - 각 친구 관계마다 채널을 두고 상태 변화 시 해당 채널에 작성하고, 친구들이 이를 구독하여 실시간 웹소켓을 통해 통지

## 4단계 마무리

- 종단 간 암호화(**E2EE)**
    
    - 왓츠앱은 메시지 전송에 있어 종단 간 암호화를 지원
    - 메시지 발신인과 수신자 이외에는 아무도 메시지 내용을 볼 수 없다는 뜻
- 캐시
    
    - 클라이언트에 이미 읽은 메시지를 캐시해 두면 서버와 주고받는 데이터 양을 줄일 수 있다.
- 로딩 속도 개선
    
    - 슬랙(Slack)은 사용자의 데이터, 채널 등을 지역적으로 분산하는 네트워크를 구축하여 앱 로딩 속도를 개선
        
    - Slack의 Flannel: 대규모 확장을 위한 애플리케이션 레벨 엣지 캐시
        
        ## 1. 배경 및 문제점
        
        ### 1.1 던바의 수(Dunbar's Number)와 Slack의 성장
        
        **던바의 수 이론**
        
        - **정의**: 인간이 유지할 수 있는 안정적인 관계의 최대 수 = 약 148명
        - **근거**: 개인 간 관계 + 타인 간의 교차 관계를 추적하는 인지적 한계
        - **역사적 사례**: 신석기 농업 마을, 영장류 무리가 이 규모로 유지됨
        
        **Slack의 현실**
        
        - 초기 설계: 수백 명 규모 팀 연결을 목표로 설계
        - 현재 상황: 수만 명 규모로 성장한 팀들 존재
        - **문제**: 초기 데이터 로딩 및 유지 기법이 확장성 한계에 도달
        
        → Slack은 원래 작은 마을 규모의 팀을 위해 설계되었으나, 현대 기업들이 수만 명으로 성장하면서 기존 아키텍처가 한계를 드러냈습니다.
        
        ---
        
        ### 1.2 기존 시스템의 문제점
        
        **기존 아키텍처 (Pre-Flannel)**
        
        - `rtm.start` API 호출로 팀 데이터 전체 스냅샷 전송
        - 포함 데이터: 사용자, 채널, 채널 멤버, 최근 메시지, DM, 그룹 DM 등
        - 클라이언트가 이 데이터로 부트스트랩 후 WebSocket으로 실시간 이벤트 수신
        
        **대규모 팀에서 발생하는 문제**
        
        |문제점|영향|
        |---|---|
        |**연결 시간 과다**|로딩 화면 대기 시간 증가, 채널 전환 시 첫 메시지 표시 지연|
        |**클라이언트 메모리 부담**|클라이언트 측에서 과도한 데이터 저장 및 처리|
        |**재연결 비용 증가**|노트북 닫았다 열 때마다 초기 연결만큼 리소스 소모|
        |**재연결 폭풍(Reconnection Storm)**|전체 오피스가 동시에 재연결 시 서버 부하 급증 → 연결 지연, 실패, 재시도 증가|
        
        ---
        
        ## 2. 해결책: Flannel
        
        ### 2.1 Flannel의 핵심 개념
        
        **설계 철학**
        
        - **지연 로딩(Lazy Loading)**: 최소한의 데이터만 미리 로드
        - **적시 로딩(Just-in-Time Loading)**: 필요할 때 채널/사용자 데이터 로드
        - **사용자 경험**: 원활한 경험 유지하면서 리소스 절약
        
        **Flannel 정의**
        
        - **역할**: 애플리케이션 레벨 캐싱 서비스
        - **위치**: 엣지 POP(Points of Presence)에 배포
        - **기능**:
            - 클라이언트 시작 시 관련 데이터 캐싱 (사용자, 채널, 봇 등)
            - 클라이언트 요청 시 쿼리 API 제공
            - **예측적 데이터 푸시**: 다음에 요청될 데이터 미리 전송
        
        **예측적 푸시 예시**
        
        - 상황: 채널에서 동료 멘션(@mention)
        - Flannel 동작
            1. 일부 클라이언트가 해당 사용자 정보를 최근에 로드하지 않았음을 감지
            2. 메시지 브로드캐스트 직전에 사용자 데이터 먼저 전송
            3. **효과**: 클라이언트가 별도로 쿼리할 필요 없음 (왕복 시간 절약)
        
        ---
        
        ### 2.2 아키텍처 변화
        
        **Pre-Flannel 아키텍처**
        
        ```
        클라이언트 → Slack 서버 (AWS 메인 리전)
                   ↓
            전체 데이터 다운로드
        
        ```
        
        **Flannel 아키텍처**
        
        ```
        클라이언트 → Flannel (엣지 POP) → Slack 서버 (AWS 메인 리전)
                   ↓                    ↓
            최소 데이터만              전체 데이터 캐싱
            + 필요 시 쿼리             + WebSocket 연결
        
        ```
        
        **연결 프로세스**
        
        1. **클라이언트 → Flannel 연결**
        2. **Flannel의 백그라운드 동작**:
            - AWS 메인 리전의 Slack 서버에서 전체 클라이언트 시작 데이터 수집
            - WebSocket 연결로 실시간 이벤트 소비하여 최신 상태 유지
        3. **Flannel → 클라이언트**: 축소된 시작 데이터만 반환
        4. **클라이언트 사용 준비 완료**
        
        → 엣지에 Flannel을 배치하여 클라이언트와의 물리적 거리를 줄이고, 메인 서버 부하를 분산시킵니다.
        
        ---
        
        ### 2.3 실제 사용 사례
        
        **Quick Switcher (빠른 전환기)**
        
        - **단축키**: Mac에서 cmd+k, Windows에서 ctrl+k
        - **동작 방식**:
            1. 연결 직후: 클라이언트는 팀 상태에 대한 최소 정보만 보유
            2. 타이핑 시작: 클라이언트가 Flannel에 자동완성 쿼리 전송
            3. 순간적으로 결과 렌더링: 드롭다운에 제안 표시
        - **유사 기능**: 메시지 바에서 사용자명 입력 시 동일한 프로세스
        
        **일관된 해싱(Consistent Hashing)**
        
        - **목적**: 팀 친화성(Team Affinity) 유지
        - **동작**: 같은 팀의 같은 네트워크 지역 사용자 → 동일한 Flannel 인스턴스
        - **효과**: 캐시 효율성 최적화 + 재연결 폭풍 시 백엔드 서버 부담 감소
        
        **설명**: 캐시 히트율을 높이기 위해 같은 팀 사용자들을 같은 Flannel 서버로 라우팅합니다.

        ## 3. 현황 및 성과
        
        ### 3.1 운영 현황 (2017년 1월 출시)
        
        **규모**
        
        - **동시 연결**: 피크 시 400만 개
        - **초당 쿼리**: 60만 개
        
        **성능 개선 사례**
        
        |팀 크기|페이로드 크기 감소|
        |---|---|
        |1,500명 팀|**7배 감소**|
        |32,000명 팀|**44배 감소**|
        
        **Team Directory 로딩 비교**
        
        - Pre-Flannel: 큰 팀에서 수 초 대기
        - Flannel 적용: 즉시 로딩 + 타이핑하며 점진적 데이터 표시
        
        **근거**: 전체 데이터를 미리 로드하지 않고 필요한 것만 가져오므로 초기 페이로드가 극적으로 감소합니다.
        
        ---
        
        ### 3.2 향후 계획
        
        ### 3.2.1 지속적인 지연 로딩 확장
        
        - **현재**: 사용자 및 채널 멤버 지연 로딩 완료
        - **다음 단계**: 채널 객체도 제거 예정
        - **모바일 확장**: 웹 클라이언트 성공 후 모바일 사용자에게도 적용
        
        ### 3.2.2 이벤트 팬아웃 Flannel로 이동
        
        **현재 구조**
        
        ```
        메시지 서버 → 각 연결된 사용자마다 Flannel에 복사본 전송
                      (N명 연결 = N개 전송)
        
        ```
        
        **개선 목표**
        
        ```
        메시지 서버 → 각 Flannel 서버에 1개씩만 전송
                      Flannel → 연결된 사용자들에게 팬아웃
        
        ```
        
        **효과**
        
        - 네트워크 대역폭 소비 대폭 감소
        - 백엔드 CPU 오버헤드 감소
        
        → 메시지, 사용자 프로필 업데이트, 사용자 상태 변경 등 모든 이벤트 타입에 적용 가능
        
        ---
        
        ### 3.2.3 Pub/Sub 모델로 전환
        
        **현재 모델 (Broadcast)**
        
        - 클라이언트가 팀의 모든 이벤트 수신
        - 포함: 모든 채널의 메시지, 사용자 프로필 업데이트, 사용자 상태 변경 등
        - **문제**: 불필요한 데이터 과다 수신
        
        **Pub/Sub 모델**
        
        - 클라이언트가 현재 뷰와 관련된 이벤트만 구독
        - 뷰 전환 시 구독 변경
        
        **실제 성과: 사용자 상태(Presence) 이벤트**
        
        - Pub/Sub 적용 후: **이벤트 수 5배 감소**
        - **그래프 결과**: 극적인 이벤트 감소 확인
        
        **향후 확장**
        
        - 더 많은 이벤트 타입을 Pub/Sub으로 전환
        - 클라이언트 성능 추가 개선 기대
        
        → 사용자가 채널 A를 보고 있을 때 채널 B~Z의 메시지를 실시간으로 받을 필요가 없습니다.

        ## 4. 핵심 교훈 및 결론
        
        ### 4.1 역사적 유추
        
        **Hutterite 정착촌 & 폴리네시아 부족**
        
        - 148명 이상으로 성장 시 분리/탐험대 파견
        - **이유**: 모든 구성원과 관계를 추적할 수 없음
        - **제한 요소**: 각 개인의 "정신적 캐시" 용량
        
        **Slack + Flannel**
        
        - 유사한 제약을 중앙 서버로 해결
        - 정보를 온디맨드로 제공하여 클라이언트 부담 감소
        - **결과**: 수십만 명 규모 팀 지원 가능
        
        ---
        
        ### 4.2 아키텍처 진화의 핵심 원칙
        
        |원칙|구현|
        |---|---|
        |**엣지 캐싱**|Flannel을 POP에 배치하여 지연 시간 최소화|
        |**지연 로딩**|필요한 데이터만 필요할 때 로드|
        |**예측적 푸시**|다음 요청을 예상하여 미리 데이터 전송|
        |**팀 친화성**|일관된 해싱으로 캐시 효율성 극대화|
        |**엣지 팬아웃**|중앙 서버 부하를 엣지로 분산|
        |**선택적 구독**|Pub/Sub으로 불필요한 데이터 전송 제거|
        
        ---
        
        ### 4.3 앞서 정리한 채팅 시스템과의 연결점
        
        **메시지 동기화 큐와 Flannel**
        
        - 앞 문서: 각 사용자의 메시지 동기화 큐에 복사
        - Flannel: 엣지에서 이벤트 팬아웃 처리 → 중앙 서버 부담 감소
        
        **여러 단말 동기화와 Pub/Sub**
        
        - 앞 문서: `cur_max_message_id`로 새 메시지 판별
        - Flannel: Pub/Sub으로 현재 활성화된 단말/뷰에만 이벤트 전송
        
        **소규모 vs 대규모 그룹**
        
        - 앞 문서: 메시지 복사 방식은 소규모 그룹(~500명)에 적합
        - Flannel: 수만 명 팀을 위한 확장 가능한 솔루션 제공
        
        Flannel은 채팅 시스템의 확장성 문제를 해결하기 위한 실제 프로덕션 솔루션으로, 앞서 본 이론적 메시지 흐름을 대규모로 구현한 사례입니다.
        
- 오류 처리
    
    - 채팅 서버 오류
        - 채팅 서버 하나에 수십만 사용자가 접속해 있는 상황
        - 서버 하나가 죽으면 서비스 탐색 기능(주키퍼 같은)이 동작하여 클라이언트에게 새로운 서버를 배정하고 다시 접속할 수 있도록 해야 함
    - 메시지 재전송
        - 재시도(retry)나 큐(queue)는 메시지의 안정적 전송을 보장하기 위해 흔히 사용되는 기법이다
