# [대규모 시스템 설계 기초] 1장) 사용자 수에 따른 규모 확장성

> 1장에서는 한 명의 사용자를 지원하는 시스템에서 시작하여, 최종적으로는 몇백만 사용자를 지원하는 시스템으로 설계함

# 🧩 단일 서버

![](https://velog.velcdn.com/images/kkimdy_12/post/acf09ada-e4fd-4fec-9f55-5c2488ef1f44/image.png)

- 모든 컴포넌트가 단 한 대의 서버에서 실행되는 간단한 구성 방식
- 웹 앱, 데이터베이스, 캐시 등이 전부 서버 한 대에서 실행됨

## 사용자 요청 흐름

1. 사용자는 도메인 이름을 이용하여 웹사이트에 접속한다. 이 접속을 위해 도메인 이름을 도메인 이름 서비스(Domain Name Service, DNS)에 질의하여 IP 주소로 변환하는 과정이 필요하다.
2. DNS 조회로 IP 주소가 반환된다.
3. 해당 IP 주소로 HTTP 요청이 전달된다.
4. 요청을 받은 웹 서버는 HTML 페이지나 JSON 형태의 응답을 반환한다.

# 🧩 데이터베이스

![](https://velog.velcdn.com/images/kkimdy_12/post/37bc6eb9-80b6-4baf-b097-fd0d4c7ad013/image.png)

여러 서버를 두어야하는 경우는?

- 하나는 웹/모바일 트래픽 처리용(웹 계층)
- 다른 하나는 데이터베이스용(데이터 계층) 위처럼 서버를 분리하면 독립적으로 확장해나갈 수 있게 됨!

## 어떤 데이터베이스를 사용할 것인가?

1. 관계형 데이터베이스 (RDBMS)
    - MySQL, ORACLE 데이터베이스, PostgreSQL, MSSQL 등
    - 자료를 테이블과 열, 칼럼으로 표현
    - 관계에 따라 여러 테이블에 있는 데이터를 조인(join)하여 합칠 수 있음
2. 비관계형 데이터베이스 (NoSQL)
    - MongoDB, CouchDB, Neo4j, Cassandra, HBase, Amazon DynamoDB 등
    - 네 부류: 키-값 저장소, 그래프 저장소, 칼럼 저장소, 문서 저장소
        - 키 벨류는 레디스 같은걸로 쓰기도 한다.
    - 조인 연산 지원 X

`보통은 RDBMS가 40년 이상 시장에서 살아남았기 때문에 적합함.`

## 비관계형 데이터베이스가 더 적합할 때는?

- 아주 낮은 응답 지연시간이 요구됨 ~~⇒ 아주 빠른 응답이 요구될 때~~
    - 실시간 채팅 메시지, 실시간 랭킹 데이터
        - 실시간 랭킹 데이터를 먹일 수 있는게 sorting 할 때 세팅할 수 있다.
- 다루는 데이터가 비정형이라 관계형 데이터가 아님
    - JSON/`문서 구조가 계속 변하는 데이터`
- 데이터를 직렬화하거나 역직렬화할 수 있기만 하면 됨
- 아주 많은 양의 데이터를 저장할 필요가 있음
    - ~~텍스트~~NoSQL은 처음부터 수평 확장 + 분산 중심으로 설계돼서

# 🧩 수직적 규모 확장 vs 수평적 규모 확장

![](https://velog.velcdn.com/images/kkimdy_12/post/b157b707-a910-4212-bd15-90ac623e1f47/image.png)

1. **수직적 규모 확장 - 스케일 업**
    
    - 서버에 고사양(더 좋은 CPU, 더 많은 RAM 등)을 추가하는 행위
    - 서버로 유입되는 트래픽 양이 적을 때 적합, 단순함
    - 하지만
        - 한계가 있음. 한 대의 서버에 CPU나 메모리를 무한대로 증설할 방법 X
        - 장애에 대한 자동복구 방안이나 다중화 방안을 제시하지 않음. 서버에 장애가 생기면 웹사이트/앱 완전히 중단됨.
    
    `그렇기 때문에 대규모 애플리케이션을 지원하는 데에는 2번 방식이 더 적절하다.`
    
2. **수평적 규모 확장 - 스케일 아웃**
    

- 더 많은 서버를 추가하여 성능을 개선하는 행위
- 각 서버가 일부 트래픽만 담당
- 로드밸런서, 샤딩 등을 사용해 요청을 분산 처리
- 스케일 아웃 & 스케일 인 (뗏다붙엿다..)
- 클라우드 환경이면 스케일 아웃부터 고려한다.

# 🧩 로드밸런서

유동적으로 늘어나고 줄어드는 서버에 대해서 사용자의 트래픽을 분산 시켜주어야 하는데, 로드밸런서를 통해 가능하다. (그렇지 않으면 웹 서버가 다운돼서 접속이 안되거나 응답 속도가 느려지니...)

![](https://velog.velcdn.com/images/kkimdy_12/post/6f0f5323-d874-4614-9ca2-efa5e2d885a7/image.png)

로드밸런서는 이름 그대로 '로드(부하)'를 '밸런싱(분산)' 시키는 장치로, 여러 대의 웹 서버가 있을 때, 트래픽 부하를 고르게 분산하는 역할을 한다.

**✨ 로드밸런서를 쓰면 뭐가 좋아?**

- 사용자는 서버 신경 쓸 필요 없음
    - 사용자는 오직 로드밸런서의 Public IP로만 접속한다. 로드밸런서는 내부적으로 여러 웹 서버의 Private IP로 요청을 나눠준다.
- Private IP 기반으로 보안 강화
    - 웹 서버들이 외부에 직접 노출되지 않고 같은 네트워크 안에서만 통신하므로 보안상 유리함.
- 장애 자동 우회 (No more failover nightmare)
    - 단일 서버 구조에선 서버 한 대가 죽으면 서비스가 그냥 끝이었음.
    - 근데 로드밸런서가 있으면:
        - 서버 1이 죽음 → 로드밸런서가 서버 2로 트래픽 자동 전송
        - 유저는 장애가 난 줄도 모름 이게 바로 고가용성(High Availability).

현재 웹 계층에 다중화를 적용하였지만, 데이터베이스에도 다중화를 적용해볼 수 있다.

## 데이터베이스 다중화

![](https://velog.velcdn.com/images/kkimdy_12/post/2c1b21a8-27b2-4103-b224-7b61b594440c/image.png)

데이터베이스 다중화는 master-slave 관계로 설정하고, 데이터 원본을 주(master) 서버에, 사본을 부(slave) 서버에 저장하는 방식이다. 각각의 역할도 다른데, 주 서버에서는 데이터의 쓰기 연산(insert, delete, update)만 지원하고 부 서버에서는 데이터의 읽기 연산(read)만 수행한다.

- 데이터의 쓰기 연산은 하나의 주 데이터베이스가 처리하는 반면, 부 데이터베이스의 병렬로 처리할 수 있는 질의(query)의 수가 늘어나게 되면서 성능이 향상된다.

또한 데이터베이스 서버 중 일부가 장애를 겪어도 다른 서버에서 처리하면 되기 때문에 안정성과 가용성이 있다. 위의 설계는 다음과 같은 상황을 감당할 수 있다.

1. 부 DB 서버 중 한 대 다운 시 → 나머지 부 DB 서버가 장애 서버를 대체
2. 주 DB 서버 다운 시 → 하나의 부 DB 서버가 주 DB 서버 역할을 수행 (프로모션)

하지만 부 데이터베이서 서버는 사본을 저장하고 있기 때문에 부 서버에서 보관하고 있는 데이터가 언제나 최신 상태가 아닐 수 있다. (다중 마스터나 원형 다중화로 해결이 가능하긴 하지만 구성이 복잡하다.)

![](https://velog.velcdn.com/images/kkimdy_12/post/fd0a304f-5098-42da-9b26-c15fbd193746/image.png)

로드밸런서와 데이터베이스 다중화를 적용한 설계의 흐름

1. 사용자는 DNS로부터 로드밸런서의 공개 IP 주소 획득
2. 로드밸런서의 공개 IP로 접속
3. 로드밸런서가 서버의 상황에 따라 서버 1 or 서버 2로 요청 전달
4. 웹서버가 부 데이터베이스에서 데이터를 읽음
5. 데이터 변경 연산 시 주 데이터베이스로 전달

# 🧩 캐시

- 값비싼 연산 결과 또는 자주 참조되는 데이터를 메모리 안에 두고, 뒤이은 요청이 보다 빨리 처리될 수 있도록 하는 저장소
- 애플리케이션의 성능능은 데이터베이스를 얼마나 자주 호출하느냐에 크게 좌우되는데, 캐시는 그런 문제를 완화할 수 있다.

## 캐시 계층

- 데이터가 잠시 보관되는 곳으로 데이터베이스보다 훨씬 빠르다.
    
- 별도의 캐시 계층을 두면 성능이 개선될 뿐 아니라 데이터베이스의 부하를 줄일 수 있고, 캐시 계층의 규모를 독립적으로 확장시키는 것도 가능해진다.
    
- 아래는 _우선 읽기 전략(read-through caching strategy)_
    
    ![](https://velog.velcdn.com/images/kkimdy_12/post/b2e76cb9-f34e-482e-98cd-e98b1c27cb2b/image.png)
    

**캐시 사용 시 유의할 점**

- 데이터 갱신은 자주 일어나지 않지만 참조는 빈번하게 일어난다면 캐시 적용을 고려해볼만 하다.
- 영속적으로 보관할 데이터를 캐시에 두는 것은 바람직하지 않다.
    - 캐시는 데이터를 휘발성 메모리에 두기 때문에, 캐시 서버가 재시작되면 모든 데이터는 사라진다.
- 만료된 데이터는 캐시에서 삭제되어야 한다.
    - 만료 기한에 대한 정책을 마련하는 것이 좋다.
    - 만료 기한이 너무 짧으면, 데이터베이스를 너무 자주 읽게 되며, 너무 길면 원본과 차이가 날 가능성이 높아진다.
- 데이터 저장소의 원본과 캐시 내의 사본 일관성을 확인해야 한다.
- 캐시 서버를 한 대만 두는 경우 해당 서버는 단일 장애 지점(SPOF)이 되어 버릴 수 있다.
- 캐시 메모리 크기는 너무 크지도 작지도 않게 적절하게 잡아야 한다. (너무 작으면 데이터가 자주 캐시에서 밀려나 캐시의 성능이 떨어지게 됨)
- 캐시가 꽉 차버리면 LRU, LFU, FIFO 같은 정책들을 사용해서 사용해야 한다. (데이터 방출 정책)

# 🧩 콘텐츠 전송 네트워크 (CDN)

- GPT가 나중에 그려줄것임 (그림)
    
    ![image.png](attachment:30889ee9-4804-4c56-a61d-288101396243:image.png)
    
- 정적 콘텐츠를 전송하는 데 쓰이는, 지리적으로 분산된 서버의 네트워크
    

> 해외 서버에 5GB 영상을 저장해두고 한국에서 다운로드하려고 하면, 물리적인 거리 때문에 패킷 라우팅 횟수가 많아지고 자연스럽게 네트워크 지연이 발생한다.

반면 CDN을 사용하면, 한국에 위치한 CDN 서버에 해당 영상이 캐싱되어 가까운 서버에서 빠르게 파일을 내려받을 수 있다.

![](https://velog.velcdn.com/images/kkimdy_12/post/f7172670-7b3b-435e-b608-dca212f2ed16/image.png)

사용자가 웹사이트를 방문하면, 그 사용자의 위치와 가장 가까운 CDN 서버가 정적 콘텐츠를 전달한다. CDN 서버가 가까울수록 먼 사람에 비해 웹사이트를 빨리 받을 수 있다. 위의 그림이 CDN을 사용함으로써 사이트 로딩 시간을 단축시킨 예시이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/be8a0d92-a5a0-4c4d-bdbb-8b4d5f633644/image.png)

만약 CDN 서버에 요청한 이미지가 없다면,

1. 원본 서버(origin)에서 파일을 가져오고
2. 해당 파일을 CDN에 저장한 뒤
3. 사용자에게 반환한다 그 이후 동일한 파일 요청은 CDN에서 직접 제공되므로 응답 시간이 크게 단축된다.

**✨고려해야하는 부분들**

1. CDN 비용 고려 CDN은 대부분 제3 사업자가 운영하는 만큼, CDN 서버에서 나가는 트래픽 양에 따라 비용이 발생한다. 그래서 자주 사용되지 않는 파일을 캐싱하면 오히려 비효율적이다. 캐시 미스가 자주 발생하면 다시 원본 서버에서 가져와야 하기 때문! (자주 사용되는 콘텐츠를 캐싱해야 한다.)
2. 캐시 만료(TTL) 관리 캐시 만료 시점을 적절히 설정하는 것도 중요하다.

- 너무 길면 → 오래된 콘텐츠를 제공할 위험
- 너무 짧으면 → 원본 서버에 자주 접근하게 되어 부하 증가 원본 서버는 CDN에 콘텐츠를 전달할 때 TTL(Time To Live) 값을 함께 주어, 캐시가 얼마나 유지될지를 설정한다.

다음은 로드밸런서 + 데이터베이스 다중화 + CDN & 캐시가 추가된 설계이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/3aba31f0-6ffb-4905-95f9-5c1c6b09fa5d/image.png)

정적 콘텐츠는 웹 서버 대신 CDN을 통해 제공되기 때문에 성능이 더 좋아지고, 캐시를 사용하면 데이터베이스에 직접 접근하는 횟수가 줄어들어 부하가 크게 감소한다.

# 🧩 무상태(stateless) 웹 계층

- 웹 계층을 수평적으로 확장하고 싶다면? 상태 정보(사용자 세션 데이터와 같은)를 웹 계층에서 제거하여야 한다. 바람직한 전략은 상태 정보를 관계형 데이터베이스나 NoSQL 같은 지속성 저장소에 보관하고, 필요할 때 가져오도록 하는 것이다. 이렇게 구성된 웹 계층을 무상태 웹 계층이라 부른다.

## 상태 정보 의존적인 아키텍처

![](https://velog.velcdn.com/images/kkimdy_12/post/ebbcaa0c-dbc8-4992-81a4-7a55b6e205bd/image.png)

사용자 A의 세션 데이터나, 프로파일 이미지는 서버 1에 의존하고 있기 때문에 HTTP 요청이 반드시 서버 1로 전송되어야 한다. 서버 2는 사용자 A에 대한 정보를 갖고있지 않기 때문에 요청이 실패할 것이다.이러한 문제를 해결하기 위하여 서버들끼리 공유하는 공유 저장소를 도입해보자.

![](https://velog.velcdn.com/images/kkimdy_12/post/abaa43a5-784f-4962-b63f-6dd1a42477f7/image.png)

이제는 상태 정보를 웹 서버에서 분리시켰기에 클라이언트의 요청은 어떤 서버로도 전달될 수 있다. 웹서버는 사용자의 상태 정보가 필요할 경우 공유 저장소에서 데이터를 가져오면 된다. 이 공유 저장소는 RDBMS일 수도 있고, Redis와 같은 캐시 시스템일수도 있고, NoSQL일 수도 있다. NoSQL이 수평적으로 확장하기 용이하기 때문에 여기에서는 NoSQL을 사용하였다.다음은 로드밸런서 + 데이터베이스 다중화 + CDN & 캐시 + 공유 저장소까지 포함한 아키텍처이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/9a327a19-199f-43e1-ae98-c7c1a1588a0b/image.png)

웹 서버에서 상태 정보를 분리시켜 무 상태 웹 계층을 갖도록 하게 된 결과, 서버의 확장이 용이해졌다. 트래픽의 양에 따라 웹 서버를 추가하거나 삭제할 수 있는 자동 규모 확장(autoscaling) 기능을 수행할 수 있게 되었다.

# 🧩 데이터 센터

전 세계 곳곳에 데이터 센터를 두면, 사용자는 가장 가까운 데이터 센터로 연결되므로 더 빠르고 안정적인 서비스를 이용할 수 있다. 이를

**지리적 라우팅(geoDNS-routing)**

이라고 부른다.

![](https://velog.velcdn.com/images/kkimdy_12/post/1d39b7da-215e-4b15-9a49-baf806edf992/image.png)

- 현재 x%의 사용자는 KR-EAST 센터로, (100-x)%의 사용자는 KR-WEST 센터로 연결된다. 만약 KR-EAST 센터에 장애가 생겨 서비스 불가 시 모든 트래픽이 KR-WEST로 몰릴 것이다.

다중 데이터 센터 아키텍처를 만들려면 조건이 필요하다.

- 트래픽 우회: 장애 상황에서 정상적인 데이터 센터로 트래픽을 효율적으로 보내야 한다.
- 데이터 동기화: 데이터 센터마다 데이터베이스가 있을 때, 장애로 인해 데이터 센터1 지역을 사용하지 못한다면, 다른 데이터센터로 트래픽을 우회한다고 하더라도 데이터 센터1에서만 사용자가 원하는 데이터가 존재하기 때문에 데이터가 없을 수 있다.
- 테스트와 배포: 웹 사이트 또는 애플리케이션을 여러 위치에서 테스트할 수 있어야 한다. 또한, 자동화된 배포 도구를 이용하여 모든 데이터 센터에 배포할 수 있어야 한다.

> 이러한 기술적 과제를 해결하면 글로벌 서비스 인프라가 완성된다. 시스템 규모가 더 커지면 컴포넌트를 분리해 독립적으로 확장할 수 있도록 만들면 된다.

# 🧩 로그, 메트릭 그리고 자동화

서비스가 성장할수록 시스템의 상태를 파악하고, 문제를 빠르게 감지하며, 반복적인 작업을 줄이는 것이 중요하다. 이를 위해서 로그, 메트릭, 자동화와 같이 서비스를 관리할 수 있는 체계가 필요하다.

- 로그 : 시스템에서 발생한 이벤트, 에러, 동작 등을 기록한 데이터이다. 문제 발생 시 사용자의 행동을 추적할 수 있다.
- 메트릭 : CPU, RAM, Traffic, Latency 등 시스템의 상태를 수치화 한 데이터이다.
- 자동화 : 배포, 인프라 설정, 장애 등 반복적이고 복잡한 작업을 자동으로 처리하는 방법이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/c41cbc8f-1c49-4fb2-b034-2307239e6602/image.png)

위는 데이터센터 아키텍처에 메시지큐와 관리 도구를 추가한 설계이다.

# 🧩 데이터베이스의 규모 확장

저장된 데이터의 양이 많아질수록 데이터베이스에 많은 부하가 걸린다. 이 때 데이터베이스 서버를 확장해야 하는데 위에서 알아보았던 스케일 업 방식과 스케일 아웃 방식으로 확장할 수 있다.

![](https://velog.velcdn.com/images/kkimdy_12/post/77a166fd-5dcf-4765-93cd-da46bed37c96/image.png)

스케일 업 방식으로도 충분히 확장할 수 있다. 스택오버플로우도 한 해의 1000명 사용자를 한 대의 데이터베이스 서버로 처리하였다. 하지만, 데이터베이스 서버 컴퓨터의 하드웨어에 CPU나 RAM을 무한하게 올릴 수는 없다. 고성능의 하드웨어일수록 가격도 비싸진다.따라서 데이터베이스를 스케일 아웃, 수평적으로 확장하는 방식을 사용해야 하는데 이를

**샤딩(sharding)**

이라고 한다. 샤딩은 데이터베이스를 샤드(shard)의 단위로 분할하여 저장하는 방법이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/0277fd71-cdbc-4161-9e33-9c9c015d1102/image.png)

위 그림은 샤드로 분할된 데이터베이스의 예시이다. 사용자 ID에 따라 사용자의 데이터가 어느 샤드에 들어갈 지 결정된다. 이렇게 샤딩을 구현할 때 어떤 것을 샤딩 키로 정하는지가 중요한 고려사항이다. 샤딩 키를 정하는 조건은 데이터를 고르게 분할할 수 있는 데이터를 키로 선택하는 것이 좋다.

- 케바케이므로 상황에 맞는 내용으로 샤딩 키를 정하는 조건을 잡아봅세 - DB에 따라 다르다!

✨ 샤딩으로 데이터베이스를 확장시키는 것은 몇 가지 복잡한 문제도 있다.

1. **데이터의 재 샤딩 문제**
    - 데이터가 너무 많아져서 한 샤드로 감당하기 어렵거나 샤드 간 데이터가 균등하지 않으면 다른 샤드에 비해 샤드의 공간 소모가 빠르게 줄어드는 샤드 소진 문제가 발생한다. 이럴 때는 샤드 계산 함수를 변경하고 데이터의 재배치가 필요하다.
2. **유명인사 문제**
    - 유명인사는 수많은 팔로워를 보유중이다. 손흥민, BTS, 봉준호, 코헤와 같은 유명인사가 같은 샤드에 저장되어있으면 해당 샤드는 수많은 읽기 연산으로 과부하가 걸린다. 따라서 샤드를 유명인사별로 하나씩 할당하거나 더 잘게 나눌 필요도 있다.
3. **조인 & 비정규화 문제**
    - 데이터베이스를 여러 샤드로 나누고 난 후 조인하기 어려운 문제가 있다. 이럴 때는 테이블을 비정규화 하여 하나의 테이블에서 질의를 처리해야 한다.

마지막으로 아키텍처에 데이터베이스 샤딩까지 결합한 설계이다.

![](https://velog.velcdn.com/images/kkimdy_12/post/12f438b2-3d3c-4af3-af80-5c9095ed8787/image.png)

# **[대규모 시스템 설계 기초] 2장) 개략적인 규모 측정**

> 개략적인 규모 측정: 보편적으로 사용되는 성능 수치 항목들을 추정해서 계산하는 것 필요한 기본기로 2의 제곱수나 응답지연(latency) 값과 가용성에 관계된 수치들의 이해가 있어야 한다.

# 🧩 2의 제곱수

분산 시스템이 다루는 데이터은 매우 크지만, 기본적으로 데이터 볼륨의 단위를 2의 제곱수로 표현한다. 최소단위인 1바이트(Byte)는 8비트로 구성된다. ASCII 문자 하나가 차지하는 메모리 크기가 바로 1바이트이다. 다음은 데이터 볼륨 단위를 2의 제곱수로 표현한 표이다.

|2의 지수|10의 지수 근사치|이름(축약)|저장 단위|표기|
|---|---|---|---|---|
|10|1천 (thousand)|킬로바이트|Kilobyte|1KB|
|20|1백만 (million)|메가바이트|Megabyte|1MB|
|30|10억 (billion)|기가바이트|Gigabyte|1GB|
|40|1조 (trillion)|테라바이트|Terabyte|1TB|
|50|1000조 (quadrillion)|페타바이트|Petabyte|1PB|

### **GB (Gigabyte)**

- **10진수(Decimal)** 기반 단위
- **1 GB = 1,000,000,000 bytes = 10^9 bytes**
- 하드디스크, SSD, USB 메모리 제조사들이 주로 사용하는 표기

### **GiB (Gibibyte)**

- **2진수(Binary)** 기반 단위
- **1 GiB = 1,073,741,824 bytes = 1024^3 bytes**
- 운영체제나 개발 환경에서 메모리, RAM 용량 계산 시 주로 사용

# 🧩 모든 프로그래머가 알아야하는 응답지연 값

구글의 제프 딘이 2010년에 공개한 컴퓨터 구현 연산의 응답지연 값이다. 이 수치로 컴퓨터의 연산 처리 속도를 가늠할 수 있다.

|연산명|시간|
|---|---|
|L1 캐시 참조|0.5ns|
|분기 예측 오류 (branch mispredict)|5ns|
|L2 캐시 참조|7ns|
|뮤텍스(mutex) 락/언락|100ns|
|주 메모리 참조|100ns|
|Zippy로 1KB 압축|10,000ns = 10µs|
|1Gbps 네트워크로 2KB 전송|20,000ns = 20µs|
|메모리에서 1MB 순차적으로 read|250,000ns = 250µs|
|같은 데이터 센터 내 메시지 왕복 지연시간|500,000ns = 500µs|
|디스크 탐색 (seek)|10,000,000ns = 10ms|
|네트워크에서 1MB 순차적으로 read|10,000,000ns = 10ms|
|디스크에서 1MB 순차적으로 read|30,000,000ns = 30ms|
|한 패킷의 캘리포니아 ↔ 네덜란드 왕복 지연시간|150,000,000ns = 150ms|

이 수치를 분석해보면 다음의 결론이 나온다.

- 메모리는 빠르지만 디스크는 아직 느리다.
- 디스크 탐색은 가능하면 피하라.
- 단순 압축 알고리즘은 빠르다.
- 인터넷으로 데이터 전송 전 가능하며 압축하라.
- 데이터 센터 간 데이터를 주고받는 데 시간이 걸린다.

# 🧩 트위터 QPS와 저장소 요구량 추정

가정

- 월간 능동 사용자는 3억 명이다
- 50%의 사용자가 트위터를 매일 사용한다
- 평균적으로 각 사용자는 매일 2건의 트윗을 올린다
- 미디어를 포함하는 트윗은 10% 정도다
- 데이터는 5년간 보관된다

추정

- 월간 능동 사용자는 3억 명이다
- 50%의 사용자가 트위터를 매일 사용한다
- 3억 * 50% = 1.5억 명

QPS(Query Per Second) 추정치

- 평균적으로 각 사용자는 매일 2건의 트윗을 올린다
- QPS = 1.5억 * 2 트윗 / 24시간 / 1분(3600초) = 약 3500
- 최대 QPS = 3500 * 2 = 약 7000
    - 트래픽은 하루 중 일정 시간대에 몰리기 때문에, 평균 QPS만으로는 부족해서 피크를 평균의 두 배로 잡는 것.

미디어 저장을 위한 저장소 요구량

- 평균 트윗 크기(avg)
    - tweet_id 에 64바이트
    - 텍스트에 140바이트
    - 미디어에 1MB
- 미디어를 포함하는 트윗은 10% 정도다
    - 1.5억 2(개) 10% * 1MB = 30TB/일
- 데이터는 5년간 보관된다
    - 30TB 365일 5년 = 약 55PB

# 🧩 팁

개략적인 규모 추정과 관련된 면접에서 가장 중요한 것은 문제를 풀어나가는 절차다. 즉, 올바른 절차를 밟느냐가 결과를 내는 것보다 중요하다.

- 근사치를 활용한 계산 : 99987 / 9.1 보다는 100,000 / 10로 수치를 간소화 하자.
- 가정들을 적어 놓자.
- 단위를 붙이자. ex ) 5가 5KB인지 5MB인지 적는 습관을 가지자.
- 계략적 규모 추정 문제는 QPS, 최대 QPS, 저장소 요구량, 캐시 요구량, 서버 수 등을 추정하는 것이다. 면접에 임하기 전에 계산하는 연습을 미리 하도록 하자.
