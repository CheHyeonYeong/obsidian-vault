# 13장 검색어 자동완성 시스템

## 1단계 문제 이해 및 설계 범위 확정

### 요구사항

- 빠른 응답 속도: 사용자가 검색어를 입력함에 따라 자동완성 검색어도 충분히 빨리 표시되어야 함 → 페이스북의 경우 시스템 응답속도는 100밀리초 이내
- 연관성: 자동완성되어 출력되는 검색어는 사용자가 입력한 단어와 연관된 곳이어야 함
- 정렬: 시스템의 계산 결과는 인기도 등의 순의 모델에 의해 정렬되어 있어야 함
- 규모 확장성: 많은 트래픽을 감당할 수 있도록 확장 가능해야 함
- 고가용성: 장애나 문제가 생겨도 시스템은 계속 사용할 수 있어야 함

### 개략적 규모 추정

- DAU: 천만 명
- 평균 한 사용자는 매일 10건의 검색 수행
- 질의할 때마다 평균적으로 20바이트의 데이터 입력 가정
- 검색창에 글자를 입력할 때마다 클라이언트는 검색어 자동완성 백엔드에 요청을 보냄
- 대략 초당 24,000건의 QPS 발생
- 최대 QPS 대략 48,000
- 질의 가운데 20%는 신규 검색어 → 매일 0.4GB의 신규 데이터가 시스템에 추가됨

### 트라이 자료구조

관계형 데이터베이스로 가장 인기 있는 다섯 개의 질의문을 골라내는 방법은 효율적이지 않으므로, **트라이(trie)**를 통해 해결함

**트라이**

문자열들을 간략하게 저장할 수 있는 자료구조

- 트리 형태의 자료구조
    
- 트리의 루트 노드는 빈 문자열을 나타냄
    
- 각 노드는 글자(character) 하나를 저장하며, 26개(해당 글자 다음에 등장할 수 있는 모든 글자의 개수)의 자식 노드를 가질 수 있음
    
- 각 트리 노드는 하나의 단어, 또는 접두어 문자열(prefix string)을 나타냄
    
- [그림] ‘tree’, ‘try’, ‘true’, ‘toy’, ‘wish’, ‘win’이 보관된 트라이

- 기본 트라이 자료구조는 노드에 문자들을 저장하며, 이용 빈도에 따라 정렬된 결과를 내놓기 위해 노드에 빈도 정보까지 저장해야 함

<aside> 💡

그렇다면 트라이로 검색어 자동완성은 어떻게 구현할까?

</aside>

**용어**

- p: 접두어(prefix)의 길이
- n: 트라이 안에 있는 노드 개수
- c: 주어진 노드의 자식 노드 개수

**가장 많이 사용된 질의어 k개를 찾는 방법**

- 해당 접두어를 표현하는 노드를 찾음 → 시간 복잡도 O(p)
    
- 해당 노드부터 시작하는 하위 트리를 탐색해 모든 유효 노드를 찾음
    
    - 유효한 검색 문자열을 구성하는 노드가 유효 노드
    - 시간 복잡도 O(c)
- 유효 노드들을 정렬하여 가장 인기 있는 검색어 k개를 찾음 → 시간 복잡도 O(clogc)
    
- [그림] k=2, 사용자가 검색창에 ‘be’를 입력했을 때 동작 과정
    
    1. 접두어 노드 ‘be’ 찾기
    2. 해당 노드부터 시작하는 하위 트리를 탐색해 모든 유효 노드를 찾음
        - [그림]의 경우 [beer: 10], [best: 35], [bet: 29]가 유효 노드에 해당
    3. 유효 노드를 정렬하여 2개만 골라냄 → [best: 35], [bet: 29]가 접두어(즉, 검색어) ‘tr’에 대해 검색된 2개의 인기 검색어
    
    ![화면 캡처 2025-12-16 161349.png](attachment:757164e6-5ad0-4a9e-905c-3989de1d77c3:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-16_161349.png)
    
- 시간 복잡도: O(p) + O(c) + O(clogc)
    
- 이 알고리즘은 최악의 경우 전체 트라이를 다 검색해야 하는 일이 생길 수 있고, 따라서 해결 방법 필요함
    

**해결 방법**

1. 접두어의 최대 길이 제한
2. 각 노드에 인기 검색어를 캐시

### 접두어 최대 길이 제한

사용자가 검색창에 긴 검색어를 입력하는 일은 드물어, p값을 작은 정수값이라 가정해도 안전함

검색어의 최대 길이를 제한한다면 “**접두어 노드를 찾는**” 단계의 시간 복잡도는 O(p) → **O(작은 상숫값) = O(1)**로 바뀔 것

### 노드에 인기 검색어 캐시

각 노드에 K개의 인기 검색어를 저장해 두면 전체 트라이를 검색하는 일 방지 가능

- 5~10개 정도의 자동완성 제안 표시면 충분함 ⇒ k는 작은 값
- [예제] 다섯 개의 질의 캐시
    - 각 노드에 인기 질의어를 캐시하면 ‘top 5’ 검색어를 질의하는 시간 복잡도를 낮출 수 있음
    - 단점: 각 노드에 질의어를 저장할 공간이 많이 필요함 ⇒ 빠른 응답속도가 아주 중요할 땐 이 정도 저장공간을 희생할 가치 O
- [그림] 개선된 트라이 구조
    - 각 노드에 가장 인기 있는 검색어 5개 저장

![화면 캡처 2025-12-16 162137.png](attachment:28a7ef8a-e5df-4431-bb58-1ef3dd0ef265:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-16_162137.png)

앞의 두 가지 최적화 기법을 적용했을 때 시간 복잡도 변화

1. 접두어 노드 찾는 시간 복잡도는 O(1)로 바뀜
2. 최고 인기 검색어 5개를 찾는 질의의 시간 복잡도도 O(1)로 바귐
    - 검색 결과가 이미 캐시되어 있기 때문에

⇒ 따라서, 최고 인기 검색어 k개를 찾는 전체 알고리즘의 시간 복잡도도 O(1).

### 질의 서비스

개략적 설계안에서 질의 서비스는 데이터베이스를 활용해 최고 인기 검색어 다섯 개를 골라냄

[그림] 비효율성 개선한 새 설계안

![화면 캡처 2025-12-16 170430.png](attachment:bb70204c-11d2-41fd-b98b-00629253f1f4:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-16_170430.png)

1. 검색 질의가 로드밸런서로 전송
2. 로드밸런서는 해당 질의를 API 서버로 보냄
3. API 서버는 트라이 캐시에서 데이터를 가져와 해당 요청에 대한 자동완성 검색어 제안 응답 구성
4. 데이터가 트라이 캐시에 없는 경우 → 데이터를 데이터베이스에서 가져와 캐시에 채움
    - 그래야 다음에 같은 접두어에 대한 질의가 오면 캐시에 보관된 데이터를 사용해 처리 가능
    - 캐시 미스는 캐시 서버의 메모리가 부족하거나 캐시 서버에 장애 있을 때 발생할 수 있음

⚡ **빨라야 하는 질의 서비스의 최적화 방안**

- AJAX 요청(request): AJAX 요청을 보내 자동완성된 검색어 목록을 가져오는데, 요청을 보내고 받기 위해 페이지를 새로고침할 필요가 없는 장점이 있음
- 브라우저 캐싱: 대부분 자동완성 검색어 제안 결과는 짧은 시간 안에 자주 바뀌지 않음 ⇒ 제안된 검색어들을 브라우저 캐시에 넣어두면 후속 질의의 결과는 해당 캐시에서 바로 가져갈 수 있음
- 데이터 샘플링(data sapling): 대규모 시스템의 경우, 모든 질의 결과를 로깅하도록 해 놓으면 CPU 자원과 저장공간을 엄청나게 소진함 ⇒ 이럴 때 쓰는 게 데이터 샘플링 기법인데 N개 요청 가운데 1개만 로깅하도록 함

### 저장소 규모 확장

트라이의 크기가 한 서버에 너무 큰 경우 대응할 수 있는 규모 확장성 문제를 논의해 보자

    1. 검색어를 보관하기 위해 두 대 서버가 필요하다면 ‘a’부터 ‘m’까지 글자로 시작하는 검색어는 첫 번째 서버에 저장하고, 나머지는 두 번째 서버에 저장
    2. 세 대 서버가 필요하다면 ‘a’부터 ‘i’까지는 첫 번째 서버, ‘j’부터 ‘r’까지는 두 번째 서버, 나머지는 세 번째 서버에 저장

        - 그럴싸해 보이지만 데이터를 각 서버에 균등하게 배분하기 불가능

### 비디오 업로드 절차

**개략적 설계안**

- 사용자
- 로드밸런서: API 서버 각각으로 고르게 요청 분산하는 역할
- API 서버: 비디오 스트리밍을 제외한 다른 모든 요청 처리
- 메타데이터 데이터베이스: 비디오의 메타데이터 보관
    - 샤딩, 다중화를 적용해 성능 및 가용성 요구사항 충족
- 메타데이터 캐시: 성능을 높이기 위해 비디오 메타데이터, 사용자 객체 캐시
- 원본 저장소: 원본 비디오를 보관할 대형 이전 파일 저장소(BLOB) 시스템
    - BLOB 저장소: 이진 데이터를 하나의 개체로 보관하는 데이터베이스 관리 시스템

![화면 캡처 2025-12-17 165053.png](attachment:3d74a580-8fc6-439f-8b32-e2c59c72bc0b:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-17_165053.png)

- 트랜스코딩 서버(transcoding server)
    - 비디오 트랜스코딩은 비디오 인코딩이라 부르기도 하는 절차로, 비디오의 포맷을 변환하는 절차
    - 단말이나 대역폭 요구사항에 맞는 최적의 비디오 스트림을 제공하기 위해 필요
- 트랜스코딩 비디오 저장소(transcoded storage): 트랜스코딩이 완료된 비디오를 저장하는 BLOB 저장소
- CDN
    - 비디오를 캐시하는 역할
    - 사용자가 재생 버튼을 누르면 비디오 스트리밍은 CDN을 통해 이뤄짐
- 트랜스코딩 완료 큐(completion queue): 비디오 트랜스코딩 완료 이벤트들을 보관할 메시지 큐
- 트랜스코딩 완료 핸들러(completion handler): 트랜스코딩 완료 큐에서 이벤트 데이터를 꺼내어 메타데이터 캐시와 데이터베이스를 갱신할 작업 서버들

### 비디오 업로드 처리

다음 두 프로세스가 병렬적으로 수행됨

1. 비디오 업로드
2. 비디오 메타데이터 갱신

**프로세스 a: 비디오 업로드**

1. 비디오를 원본 저장소에 업로드
2. 트랜스코딩 서버는 원본 저장소에서 해당 비디오를 가져와 트랜스코딩 시작
3. 트랜스 코딩이 완료되면 아래 두 절차 병렬적 수행
    1. 완료된 비디오를 트랜스코딩 비디오 저장소로 업로드
    2. 트랜스코딩 완료 이벤트를 트랜스코딩 완료 큐에 넣음
        1. 트랜스코딩이 끝난 비디오를 CDN에 올림
        2. 완료 핸들러가 이벤트 데이터를 큐에서 꺼냄
        3. 완료 헨들러가 메타데이터 데이터베이스와 캐시를 갱신
4. API 서버가 단말에게 비디오 업로드가 끝나 스트리밍 준비가 되었음을 알림

**프로세스 b: 비디오 메타데이터 갱신**

- 원본 저장소에 파일이 업로드되는 동안, 단말은 병렬적으로 비디오 메타데이터 갱신 요청을 API 서버에 보냄
- 이 요청에 포함된 메타데이터에는 파일 이름, 크기, 포맷 등의 정보가 들어있음
- API 서버는 이 정보로 메타데이터 캐시와 데이터베이스를 업데이트함

### 비디오 스트리밍 절차

- 스트리밍: 장치가 원격지의 비디오로부터 지속적으로 비디오 스트림을 전송 받아 영상을 재상하는 것을 의미함

**스트리밍 프로토콜**

- 비디오 스트리밍을 위해 데이터를 전송할 때 쓰이는 표준화된 통신 방법
- 종류
    - MPEG-DASD
    - 애플 HLS
    - 마이크로소프트 스무드 스트리밍
    - 어도비 HTTP 동적 스트리밍
- 프로토콜마다 지원하는 비디오 인코딩, 플레이어가 다름
- 비디오 스트리밍 서비스를 설계할 땐 서비스 용례에 맞는 프로토콜을 잘 골라야 함

**개략적 설계안**

- 비디오는 CDN에서 바로 스트리밍됨
- 사용자 단말에 가장 가까운 CDN 에지 서버가 비디오 전송 담당 ⇒ 전송 지연 아주 낮음

### 비디오 트랜스코딩

비디오를 녹화하면 단말에 저장됨

⇒ 이 비디오가 다른 단말에도 순조롭게 재생되려면 다른 단말도 호환되는 비트레이트(bitrate)와 포맷으로 저장되어야 함

- 비트레이트: 비디오를 구성하는 비트가 얼마나 빨리 처리되어야 하는지 나타내는 단위
- 비트레이트 높은 비디오: 일반적으로 고화질 비디오
    - 정상 재생을 위해서는 높은 성능의 컴퓨팅 파워, 빠른 인터넷 회신 속도 필요

**비디오 트랜스코딩이 중요한 이유**

- 가공되지 않은 원본 비디오(raw video)는 저장 공간을 많이 차지함
- 상당수의 단말, 브라우저는 특정 종류의 비디오 포맷만 지원 ⇒ 호환성 문제 해결을 위해 하나의 비디오를 여러 포맷으로 인코딩해 두는 게 바람직함
- 끊김 없는 고화질 비디오 재생 보장을 위해 ⇒ 네트워크 대역폭이 충분하지 않다면 저화질, 충분하다면 고화질 비디오를 보내는 게 바람직
- 모바일 단말의 경우 네트워크 상황이 수시로 달라질 수 있음 ⇒ 비디오가 끊김 없이 재생되도록 하기 위해 비디오 화질을 자동 or 수동 변경할 수 있게 하는 게 바람직

인코딩 포맷

- 컨테이너(container): 비디오 파일, 오디오, 메타데이터를 담는 바구니
    - .avi, .mov, .mp4 같은 파일 확장자로 컨테이너 포맷을 알 수 있음
- 코덱(codec): 비디오 화질은 보존하면서 파일 크기를 줄일 목적으로 고안된 압축 및 압축 해제 알고리즘
    - 많이 사용되는 비디오 코덱: H.264, VP9, HEVC 등

### 유향 비순환 그래프(DAG) 모델

- 각기 다른 유형의 비디오 프로세싱 파이프라인을 지원하면서,
- 처리 과정의 병렬성을 높이기 위해서,

⇒ 적절한 수준의 추상화를 도입해 클라이언트 프로그래머로 하여금 실행할 작업(task)을 손수 정의할 수 있도록 해야 함

**페이스북 스트리밍 비디오 엔진: 유향 비순환 그래프(DAG)**

- 작업을 단계별로 배열 가능하게 함 ⇒ 해당 작업들이 순차적 or 병렬적으로 실행될 수 있게 함
    
- 본 설계안도 DAG 모델을 도입해 유연성, 병렬성을 달성할 것
    
- [그림] 채택한 DAG 모델

### 비디오 트랜스코딩 아키텍처

[그림] 클라우드 서비스를 활용한 비디오 트랜스코딩 아키텍처

**전처리기**

1. 비디오 분할(video splitting)
    - 비디오 스트림을 GOP(Group of Pictures)라고 불리는 단위로 쪼갬
    - GOP: 특정 순서로 배열된 프레임 그룹
    - 하나의 GOP는 독립적으로 재생 가능하며, 길이는 보통 몇 초 정도
    - 어떤 종류의 오래된 단말, 브라우저는 GOP 단위의 비디오 분할을 지원하지 않음 → 이럴 경우 전처리기가 비디오 분할을 대신 함
2. DAG 생성
    - 클라이언트 프로그래머가 작성한 설정 파일에 따라 DAG를 만들어냄
3. 데이터 캐시
    - 전처리기는 분할된 비디오의 캐시이기도 함
    - 안정성을 높이기 위해 전처리기는 GOP와 메타데이터를 임시 저장소에 보관
    - 비디오 인코딩이 실패하면 시스템은 보관된 데이터를 활용해 인코딩 재개

**DAG 스케줄러**

- DAG 그래프를 몇 개 단계(stage)로 분할한 다음 그 각각을 자원 관리자의 작업 큐(task queue)에 집어넣음
    
- [그림] DAG 스케줄러 동작 예시
    
    ![화면 캡처 2025-12-17 192247.png](attachment:823db9cc-08bf-40c8-be78-906df057cee6:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-17_192247.png)
    
    - 하나의 DAG 그래프를 2개의 작업 단계(stage)로 쪼갠 사례
    - 1단계: 비디오, 오디오, 메타데이터 분리
    - 2단계: 해당 비디오 파일, 오디오 파일 인코딩, 섬네일 추출

**자원 관리자**

- 자원 배분을 효과적으로 수행하는 역할 담당
    
- [그림] 구성: 세 개의 큐와 작업 스케줄러(task scheduler)
    
    - 작업 큐: 실행할 작업이 보관되어 있는 우선순위 큐
    - 작업 서버 큐: 작업 서버의 가용 상태 정보 보관된 우선순위 큐
    - 실행 큐: 현재 실행 중인 작업 및 작업 서버 정보가 보관된 큐
    - 작업 스케줄러: 최적의 작업/서버 조합을 골라, 해당 작업 서버가 작업을 수행하도록 지시하는 역할
    
    ![화면 캡처 2025-12-17 192638.png](attachment:bbff71f4-c1fb-4a83-9f96-c1aae7ea3030:%ED%99%94%EB%A9%B4_%EC%BA%A1%EC%B2%98_2025-12-17_192638.png)
    
- 작업 관리자 동작 과정
    
    - 작업 관리자가 작업 큐에서 가장 높은 우선순위 작업 꺼냄
    - 작업 관리자가 해당 작업 실행 적합한 작업 서버 고름
    - 작업 스케줄러가 해당 작업 서버에게 작업 실행 지시
    - 작업 스케줄러가 해당 작업이 어떤 서버에게 할당되었는지 관한 정보를 실행 큐에 넣음
    - 작업 스케줄러가 작업이 완료되면 해당 작업을 실행 큐에서 제거

### 속도 최적화

1. **비디오 병렬 업로드**
    
    - 원본 비디오를 작은 GOP들로 분할하여 병렬적으로 업로드하면 일부가 실패해도 빠르게 업로드 재개 가능
    - 따라서 비디오를 GOP 경계에 맞춰 분할하는 작업을 단말이 수행하면 업로드 속도를 높일 수 있음
2. **업로드 센터를 사용자 근거리에 지정**
    
    - 업로드 센터를 여러 곳에 둬서 업로드 속도 개선
3. **모든 절차 병렬화**
    
    - 낮은 응답지연을 달성하는 방법 중 하나는 느슨하게 결합된 시스템을 만들어서 병렬성을 높이는 것
    - 설계안을 변경해서 비디오를 원본 저장소에서 CDN으로 옮기는 절차를 알아보자

        2. API 서버는 미리 사인된 URL을 돌려줌

1. [그림] 인기 비디오는 CDN을 통해 재생하되 다른 비디오는 비디오 서버를 통해 재생하기

### 오류 처리

**시스템 오류**

- 회복 가능 오류
    - 특정 비디오 세그먼트를 트랜스코딩하다 실패했다든가 하는 오류
    - 일반적으로 몇 번 재시도하면 해결
    - 계속 실패하고 복구 어려움 → 클라이언트에게 적절한 오류 코드를 반환해야 함

    - 비디오 포맷이 잘못되었다거나 하는 오류
    - 시스템은 해당 비디오에 대한 작업을 중단하고 클라이언트에게 적절한 오류 코드 반환해야 함
