
1. 파티셔닝과 복제  
	데이터셋이 매우 크거나 질의 처리량이 높을 때 데이터를 파티션으로 쪼개는 작업(샤딩)이 필요  
	**보통 복제와 파티셔닝을 함께 적용하여 각 파티션의 복사본을 여러 노드에 저장하며, 이를 통해 내결함성을 보장**  
	각 노드는 특정 파티션에 대해서는 리더 역할을 수행하고, 다른 파티션에 대해서는 팔로워 역할을 수행할 수 있음  
	- 보통은 확장성을 위해서 함
	- 비공유 클러스터-> 다른 파티션은 타 노드에 저장될 수 있음
		- 비공유(shared nothing)의 의미는 몰까?.?
		- 오잉 샤딩은 모두 비공유 클러스터라고 보나? (아 당연히도 맞겠구나 머쓱)
		- 질의는 모 여기저기도 가능하고 한 곳 몰빵도 가능
	- 파티셔닝 지원 DB : #랜덤_논스톱_SQL #NoSQL
2. 키-값 데이터 파티셔닝  
	- 파티셔닝 목적: 데이터와 질의 부하를 노드 사이에 고르게 분산시키기  
	- 쏠림(skewed): 특정 파티션에 부하가 몰리는 현상  
	- 핫스팟: 부하가 높은 파티션  
	- 이를 회피하기 위한 파티셔닝 방법 => 키 범위 기준 파티셔닝, 키의 해시값 기준 파티셔닝  
	1.  키 범위 기준 파티셔닝  (= 레인지 샤딩)
		종이 백과사전처럼 각 파티션에 연속된 키 범위(최솟값~최댓값)를 할당  
		장점: 키가 정렬되어 있어 범위 스캔에 유리  
		단점: 특정 접근 패턴이 핫스팟을 유발할 수 있음 (예: 타임스탬프 기반 데이터의 쓰기 쏠림)   
	2. 키의 해시값 기준 파티셔닝  
		쏠림과 핫스팟 위험을 줄이기 위해 해시 함수를 사용하여 키의 파티션을 결정  
		장점: 이데이터를 균일하게 분산시키는 데 좋음  
		단점: 범위 질의 효율성이 떨어짐  
		카산드라는 복합 기본키를 사용하여 첫 번째 칼럼은 해싱(파티션 결정)에 쓰고, 남은 칼럼은 데이터 정렬(범위 스캔용)에 사용하는 타협안을 제시   
		
		- 모듈러 샤딩 (hash % N)
		- Consistent Hashing
		- 기타 해시 분산 전략
			- 카산드라, 몽고DB -> MD5
			- 볼트모트 : 파울로 놀 보 함수
			- 엥 이게 그냥 프로그램 내장 함수랑 머가 다름? -> 해시 테이블에 사용하지 파티셔닝에 적합하지 않음;;
				- java 의 object.hastCode() 랑 ruby의 oubct#hash는 다른 프로세스에 다른 해시값 반환함
		- 일관성 해싱
		- 엥 그럼 로그가 멀까...
			- ES :  변경사항 저장용 도그, 약간 편하다 insert에 이점이 있다
				- LSM은 log를 쌓기 편하다, => 이게 매트릭 찍기 위함이 아님
				- 로그의 포맷
					- string 
					- json - 좋아보임 key-value 형식
					- ?? 하나 더 있음
				- 그럼 멀 로그로 쌓냐?
					- access log : 쿼리 body, slo 응답 성공류
					- server log
					- http 응답시간 (p99)
					-  감사로그
			- 매트릭 : 어떤 시점의 상태 (서버나 메모리 등의) 정보를 갖고 있음
				- 요구사항에 따라 append 되는데 이게 일주일이나 하루 등의 시간과 관련이 있으며 시계열 데이터로 만들어 줌
					- 프로메테우스 : tsdb(시계열 데이터 베이스)
						- 일주일치, 한달치 쿼리 이상 감지 등에 필요함
						- segment tree 로 만들고 있음
						- 모든 데이터를 다 들고 있지 않고 다운 샘플링을 하면서 확률기반으로 최적화할 수 있다
						- 머 고릴라 알고리즘으로 delta 압축이 있다!
	3. 쏠린 작업부하와 핫스팟 완화 
		- 극단적으로 동일한 키에 부하가 몰리는 경우(예: 유명인의 소셜 미디어 활동),  
		- 애플리케이션 단에서 키의 시작이나 끝에 임의의 숫자를 붙여 부하를 분산시키는 기법을 사용할 수 있음  
		- 질문 : 파티셔닝 시 key가 pk나 보조키를 사용할 수 있다
			- 이 경우 trade off가 게시글-댓글 사이의 fk/pk 중 뭘 파티셔닝 키로 해야할까?
				- fk : 핫키 문제가 있다
				- pk : 샤딩 시 모든 db를 서치하는 문제가 있다
				- 이 때 기준을 어떻게 두고 검색을 해야할까?
	  
3.  파티셔닝과 보조 색인  
	- 보조 색인은 보통 레코드를 유일하게 식별하는 용도가 아니라 특정 값을 가진 항목을 검색하는 수단  
	- 파티셔닝에 보조 색인이 얹어질 경우 복잡해지는데 그래도 사용할 경우 아래와 같은 방법을 사용함  
	- 보조색인 -> 솔라나 엘라스틱서치 같은 검색서버의 존재 이유라고 한다
	1.  문서 기준 보조 색인 파티셔닝(지역 색인)  
		각 파티션이 자신의 보조 색인을 독립적으로 유지  
		쓰기 시: 해당 파티션만 다룸  
		읽기 시: 모든 파티션에 질의를 보내 결과를 모아야 하는 스캐터/개더
			- 꼬리 지연 시간 증폭이 발생하기 쉽다 - 실전 백분위 참고
		(scatter/gather) 방식이 필요하여 비용이 많이 들 수 있음  
		- 전역색인과 지역색인이 있음
	2.  용어 기준 보조 색인 파티셔닝(전역 색인)  
		- 용어를 기준으로 파티셔닝
			- 용어는 전문색인에서 나왔는데 문서에 등장하는 모든 단어가 용어다
		모든 파티션의 데이터를 담당하는 전역 색인을 만들고, 이 색인 자체도 파티셔닝  
		쓰기 시: 느리고 복잡함  
		읽기 시: 원하는 용어가 포함된 파티션으로만 요청 가능해 읽기 효율적  
  
4. 파티션 재균형화 (Rebalancing)  
	시간이 지나 부하가 증가하거나 노드에 장애가 발생하면 데이터와 요청을 한 노드에서 다른 노드로 옮기는 과정이 필요  
	  
	- 재균형화 전략  
		1. 해시값에 모드(mod) N 연산: 노드 개수(N)가 바뀌면 대부분의 키가 이동해야 하므로 재균형화 비용이 지나치게 큼 (하지 마세요)  
		2. 파티션 개수 고정:  
			파티션을 노드 대수보다 많이 만들고 각 노드에 여러 파티션을 할당  
			노드가 추가되면 기존 노드에서 파티션 몇 개를 새 노드로 이동시킴  
		3. 동적 파티셔닝: 데이터 양에 따라 파티션을 쪼개거나 합침 (HBase, 몽고DB 등)  
		4. 노드 비례 파티셔닝: 노드당 할당되는 파티션 개수를 고정하여, 노드 대수가 늘어나면 개별 파티션 크기가 작아지도록 유지 (카산드라)  
		- 운영: 완전 자동 재균형화는 편리하지만 예측하기 어렵고 연쇄 장애를 일으킬 위험이 있어, 과정에 사람이 개입하는 것이 좋을 수도 있음  
5. 요청 라우팅  
	클라이언트의 요청을 올바른 파티션 노드로 전달하는 서비스 찾기(service discovery) 문제  
	- 설정 서버 구현에 의존하고 몽고스 데몬을 라우팅 계층으로
	- 가십 프로토콜을 사용함 (카산드라 리악)
	- 목시라는 라우팅 계층을 설정함
	주요 접근법  
	1. 클라이언트가 아무 노드에나 접속하고 노드가 요청을 전달함.  
	2. 모든 요청을 라우팅 계층(파티션 인지 로드 밸런서)으로 먼저 보냄.  
	3. 클라이언트가 직접 파티션 할당 정보를 파악하고 해당 노드로 접속함.  
	  
	이러한 변경 사항을 관리하기 위해 많은 분산 시스템은 주키퍼(ZooKeeper) 같은 별도의 코디네이션 서비스를 사용하여 파티션 할당 정보를 추적함
