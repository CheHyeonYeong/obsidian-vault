# [1교시] 패러다임의 전환: 껍데기는 가라 (50분)

## 0. 오프닝: 자바의 대격변 (10분)
### 🥖 비유: 붕어빵 기계 vs 붕어빵
* **과거 (OOP):** "기능(행동)" 하나를 전달하려고 해도 거대한 클래스(기계)와 상속(설치)이 필요했음.
* **현재 (FP):** "그냥 **'기능'**만 줘!" (함수가 일급 시민으로 승격됨)
* **왜 바뀌었나?** 멀티코어 프로세서 시대에 데이터를 안전하고 병렬적으로 처리하기 위해 **'불변성'**과 **'순수 함수'**가 필요해짐.

## 1. 람다(Lambda)와 익명 클래스 (Item 42) (15분)
### 📝 [실습 1] 코드로 느껴보기
아래 코드를 람다로 바꾸는 과정을 라이브로 보여주세요.

```java
// Step 1: 익명 클래스 (지저분함)
Collections.sort(list, new Comparator<String>() {
    public int compare(String s1, String s2) {
        return Integer.compare(s1.length(), s2.length());
    }
});

// Step 2: 람다 도입
Collections.sort(list, (s1, s2) -> Integer.compare(s1.length(), s2.length()));

// Step 3: 리스트 자체 API 활용 (자바 8+)
list.sort((s1, s2) -> Integer.compare(s1.length(), s2.length()));

// Step 4: 메서드 참조 (Comparator.comparingInt) - Item 43 예고
list.sort(comparingInt(String::length));
```

### ⚠️ 람다 사용 규칙 (중요!)
1. **타입 추론:** 컴파일러가 문맥을 보고 타입을 맞춥니다. (`String s1` -> `s1`)
2. **제약:** 추상 메서드가 **단 하나**인 인터페이스(`@FunctionalInterface`)만 람다로 변신 가능.
3. **주의:** 람다는 문서화가 안 됩니다. **3줄 넘어가면 메서드로 빼세요.**

## 2. 메서드 참조 (Item 43) (10분)
### "그거(::) 써!"
* 람다가 `(x) -> x` 처럼 단순 전달만 할 때 사용.
* **퀴즈:** 다음 람다를 메서드 참조로 바꿔보세요.
    * `s -> System.out.println(s)`  👉  `System.out::println`
    * `(a, b) -> Math.max(a, b)`  👉  `Math::max`
    * `() -> new ArrayList<>()`  👉  `ArrayList::new`

## 3. 표준 함수형 인터페이스 (Item 44) (15분)
### "자바가 차려놓은 43첩 밥상"
직접 인터페이스 만들지 말고 이거 쓰세요.

1. **Predicate (판별사):** `T -> boolean` (입구컷 담당)
2. **Function (변환사):** `T -> R` (재료 손질 담당)
3. **Consumer (먹보):** `T -> void` (잔반 처리 담당)
4. **Supplier (공급책):** `() -> T` (재료 납품 담당)

# [2교시] 스트림(Stream): 데이터 파이프라인 구축 (50분)

## 1. 스트림의 철학 (Item 45) (15분)
### "반복문(Loop) vs 스트림(Stream)"
* **Loop (How):** "1번부터 끝까지 돌면서, 만약 짝수면, 리스트에 담아." (명령형)
* **Stream (What):** "짝수만 골라서(filter), 리스트로 줘(collect)." (선언형)

### 🌊 스트림의 3단계 (라이프사이클)
1. **생성:** `stream()` (수도꼭지 틀기)
2. **중간 연산:** `filter`, `map`, `sorted` (파이프 연결 - **Lazy함!**)
    * *Lazy?* 최종 연산이 오기 전까진 실행되지 않음. (효율성 끝판왕)
3. **최종 연산:** `collect`, `forEach`, `count` (물 틀기 - **Eager**)

## 2. 부작용 없는 함수 (Item 46) (20분)
### 🚨 [경고] forEach는 '생각'하는 곳이 아니다!
* 가장 많이 하는 실수: `forEach` 안에서 `if`문 쓰고 리스트에 `add` 하기.
* **규칙:** `forEach`는 스트림 계산 결과를 **보고(Report)**할 때만 써라.

### ✨ Collectors 심층 탐구 (여기가 하이라이트)
스트림의 꽃은 `collect`입니다. 이것만 잘 써도 '고수' 소리 듣습니다.

1. **toList, toSet:** 기본 중의 기본.
2. **joining:** 문자열 합치기 (반복문으로 String 더하기 이제 그만!)
3. **groupingBy:** SQL의 `GROUP BY`와 같음.
    ```java
    // 피자를 사이즈별로 그룹핑하기
    Map<Size, List<Pizza>> bySize = pizzas.stream()
        .collect(groupingBy(Pizza::getSize));
    ```
4. **toMap:** 리스트를 맵으로 변환 (Key 중복 주의 전략 설명 필요)

## 3. 중간 실습 (15분)
* **미션:** "문자열 리스트에서 길이가 5 이상인 단어들만 골라 대문자로 변환한 뒤, 쉼표(,)로 연결해서 출력하시오."
* **정답 코드:**
```java
String result = words.stream()
    .filter(w -> w.length() >= 5)
    .map(String::toUpperCase)
    .collect(Collectors.joining(", "));
System.out.println(result);
```

# [3교시] 실전 응용과 성능 (50분)

## 1. 반환 타입 설계 (Item 47) (10분)
### "Stream을 리턴하지 마세요"
* 스트림은 한 번 쓰면 닫힙니다(재사용 불가).
* 사용자는 `for-each`를 쓰고 싶을 수도 있습니다.
* **결론:** `Collection` (List, Set)으로 반환하는 게 가장 친절합니다.

## 2. 병렬화의 함정 (Item 48) (15분)
### "parallel() 누르면 빨라지나요?" -> "아니요, 터집니다."
* 병렬화는 오버헤드(비용)가 있습니다. (나누고 + 합치고 + 스레드 관리)
* **언제 쓰는가?**
    1. 데이터가 엄청 많을 때 (수십만 건 이상).
    2. `ArrayList`처럼 쪼개기 쉬운 자료구조일 때.
    3. 독립적인 연산일 때.
* **절대 금지:** `iterate`, `limit` 등 순서가 중요한 작업에 병렬화 걸면 성능 박살남.

## 3. 스트림 지옥(Stream Hell) 리팩토링 (Item 45) (15분)
### 📝 [실습 2] 이 코드를 구원해주세요
* 복잡하게 꼬인 스트림 코드를 보여주고, **"이게 뭘 하는 코드일까요?"** 퀴즈 내기.
* **해결책:**
    1. 람다를 메서드 참조로 바꾼다.
    2. 복잡한 `filter` 조건은 별도 메서드(`isValidUser`)로 뺀다.
    3. 도저히 안 되면 `for-loop`로 돌아간다. (부끄러운 게 아님!)
